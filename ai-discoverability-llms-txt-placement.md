## Context

The Swift Institute ecosystem comprises 60+ Swift packages distributed across
17 GitHub orgs (per [feedback_superrepo_terminology]) plus a public website at
`swift-institute.org` (DocC catalog). As packages launch, retrieval-time AI
discoverability — whether ChatGPT, Claude, Perplexity, Cursor, Copilot, and
similar assistants surface the ecosystem when users ask Swift questions —
becomes a publication-strategy lever distinct from training-time inclusion.

The `llms.txt` convention (proposed by AnswerDotAI, llmstxt.org) and its
sister proposal `LLMS.md` (llmspec/llms-spec) are emerging "robots.txt for
LLMs" files — curated, machine-readable indexes that AI agents can fetch to
get a project's canonical surface without scraping HTML.

The ecosystem must decide where these files live: only on the public website,
also at the repo level, also at the GitHub org level, or some combination.
The decision is precedent-setting because subsequent package launches inherit
whatever shape we land here, and reversing later requires editing 60+ repos.

**Trigger**: User question 2026-05-01 about optimizing for AI findability.

**Constraints**:
- 17 ecosystem orgs in three tiers (top-level / org-of-orgs / leaf), per
  `feedback_superrepo_terminology`. Layer-level "containers" are not real
  surfaces — `swift-primitives/` is a clone-mirror directory, not a website
  or org.
- Package launches are gated by `feedback_no_public_or_tag_without_explicit_yes`
  — any change touching public-facing repo state needs per-action authorization.
- The user has explicitly requested low-friction defaults that don't add
  per-package launch ceremony beyond what already exists.

## Question

Where should `llms.txt` (and any companion file like `llms-full.txt` or
`LLMS.md`) live across the Swift Institute ecosystem?

Sub-questions:

1. Single file at `swift-institute.org/llms.txt`, or path-scoped per package
   on the website?
2. Per-repo `LLMS.md` at each package's GitHub root?
3. Per-org-page presence (GitHub `.github/profile/README.md` or equivalent)?
4. Companion `llms-full.txt` — yes/no, and where?

## Analysis

### Specification status (verified 2026-05-01)

Two distinct conventions, both pre-1.0:

| Convention | Source | Location | Status |
|------------|--------|----------|--------|
| `llms.txt` | llmstxt.org / AnswerDotAI | Website root `/llms.txt`; subpaths "optionally" allowed | Active proposal; ~1 year old; uneven adoption |
| `LLMS.md` | llmspec/llms-spec | Repo root | Draft v0.1; proposed for Copilot, Cursor, Claude Code, Gemini CLI |
| `llms-full.txt` | Community convention | Website root, alongside `llms.txt` | NOT in original spec; convention only |
| `llms-ctx.txt` / `llms-ctx-full.txt` | llmstxt.org | Generated by `llms_txt2ctx` tool | Spec-defined processing outputs, not authored files |

**Verified claim**: `llms-full.txt` is a community convention layered on top
of the original spec, not a spec-defined file. The spec's own "expanded
context" outputs are `llms-ctx.txt` / `llms-ctx-full.txt`, but the
community has converged on `llms-full.txt` as the de facto naming for a
hand-curated full-corpus single file. Both names appear in production
(Anthropic, Stripe, Cloudflare, Perplexity, Mintlify-hosted docs). Source:
[llmstxt.org spec](https://llmstxt.org), verified by WebFetch 2026-05-01.

**Verified claim**: GitHub Issue #18 on AnswerDotAI/llms-txt asks about
multi-property orgs (e.g., Replicate's per-model paths). Spec author has
not provided maintainer guidance in the issue thread (verified 2026-05-01);
practical examples in the wild use path-scoped llms.txt for multi-tenant
docs sites. Source:
[AnswerDotAI/llms-txt#18](https://github.com/AnswerDotAI/llms-txt/issues/18).

**Unverified claim, flagged**: actual crawler behavior (which AI assistants
fetch llms.txt at retrieval time) is poorly documented publicly. Adoption is
visible on the publishing side; the consuming side is opaque. This research
doc treats publishing as cheap insurance against a non-zero probability of
fetching, NOT as a high-confidence claim that all four major assistants
fetch it today. Re-evaluate in 6 months.

### Consumer surfaces

AI agents reach Swift Institute content through three distinct channels.
The placement question reduces to "which file at which channel":

| Channel | What it fetches | Right file |
|---------|-----------------|------------|
| Web search (ChatGPT, Claude, Perplexity at retrieval time) | HTTPS GETs to public domains; follows links | `llms.txt` at website root |
| Code agents (Copilot, Cursor, Claude Code) | Clone/index git repos; read README, docs/, source | `LLMS.md` (or rich README) at repo root |
| Training crawlers (CommonCrawl, etc., for next-gen models) | Both above, plus broader web | Both files; structured DocC HTML |

**Layer tier is not a channel**. There is no `swift-primitives.com`, no
GitHub `swift-primitives` org page that crawlers treat as authoritative for
ecosystem context. The org page is a list of repos. Layer-level llms.txt
would have nowhere to live.

### Options

#### Option A: swift-institute.org/llms.txt only (minimum viable)

Single `llms.txt` at the org website. Curated index pointing to:
- DocC homepage
- Each launched package's GitHub repo + DocC URL
- Architecture / Layers documents
- Blog (when public)
- swift-evolution proposals (when public)

**Pros**:
- Single source of truth; one file to maintain.
- Follows spec literal: file at website root.
- Cheap: ~1 file, ~5 minutes to author initial version, append on each launch.
- Low blast radius: visible only when website is updated.

**Cons**:
- Misses code-agent channel entirely. Cursor/Copilot/Claude Code working in
  a cloned package repo see no curated context beyond README.
- No per-package canonical entry-point file; an agent looking at
  `swift-buffer-primitives` repo gets no signal that this is part of an
  ecosystem unless README spells it out.

#### Option B: Per-package LLMS.md only

Every package gets `LLMS.md` at its repo root. No org-website file.

**Pros**:
- Per-repo context for code agents.
- Each package self-describes for repo-level inspection.

**Cons**:
- 60+ files to maintain; 60+ launches gain ceremony.
- Misses web-search channel — when a user asks ChatGPT "is there a Swift
  package for X?", retrieval starts on the web, not in GitHub repos.
- LLMS.md spec is draft v0.1; committing 60+ files to a pre-1.0 spec is
  premature.
- `feedback_readme_evaluator_audience` already enforces high-quality READMEs
  optimized for evaluators; LLMS.md duplicates much of that work.

#### Option C: Path-scoped llms.txt on swift-institute.org

`swift-institute.org/llms.txt` (ecosystem index) plus
`swift-institute.org/swift-buffer-primitives/llms.txt` etc. (per-package).

**Pros**:
- Per-package granularity at the website level.
- Follows the Replicate-style multi-property pattern in the wild.

**Cons**:
- swift-institute.org doesn't host per-package content — that lives at GitHub
  and per-package DocC subdomains/subpaths. Path-scoped llms.txt at the org
  site would require the org site to mirror per-package URLs it doesn't
  otherwise own.
- Spec author has not endorsed this pattern (per Issue #18); using a
  not-yet-blessed pattern as default is fragile.

#### Option D: Layered — website llms.txt + selective LLMS.md (recommended)

Two-tier strategy:

1. **Website tier** (swift-institute.org):
   - `swift-institute.org/llms.txt` — ecosystem index. Ships with first
     public launch. Maintained as packages land.
   - `swift-institute.org/llms-full.txt` — expanded ecosystem corpus
     (Architecture, Layers, package catalog with one-line descriptions).
     Ships alongside.

2. **Repo tier** (selective, deferred):
   - **Do not** add `LLMS.md` to packages by default while spec is draft v0.1.
   - **Existing READMEs already serve this role** for code agents that read
     repo files (they're optimized for evaluators per
     `feedback_readme_evaluator_audience`).
   - **Re-evaluate** when LLMS.md spec reaches 1.0 OR when 2+ major code-agent
     vendors document fetching it (per [RES-018]'s "two consumers" hurdle
     applied to convention adoption).

3. **Org tier**: skip. GitHub org pages are not a recognized AI-discovery
   surface. A short `.github/profile/README.md` mentioning the ecosystem
   and linking to `swift-institute.org` is sufficient and routine.

**Pros**:
- Hits both web-search and code-agent channels (the latter via existing
  README discipline).
- Low maintenance: 2 website files, growing by ~one line per launch.
- Defers per-repo file proliferation until the convention earns the
  commitment.
- Reversible: if the spec evolves, the ecosystem-wide change is two files,
  not 60.

**Cons**:
- Does not preempt LLMS.md adoption; if the convention takes off in 2026
  faster than expected, adding 60+ files becomes a coordinated cohort move.
  Mitigated by the README quality bar already in place.

#### Option E: Layered + per-repo LLMS.md from day one

Same as Option D but commit to LLMS.md per repo from launch.

**Pros**:
- Ahead of convention adoption curve.

**Cons**:
- Pre-1.0 spec lock-in.
- Per-launch ceremony increase, against the constraint that launches are
  already gated.
- Duplicates README content under a different filename.
- [RES-018] hurdle: only one consumer (the LLMS.md spec authors / a few
  code-agent vendors aspirationally); insufficient evidence for the
  ecosystem-wide commit.

### Comparison

| Criterion | A: website only | B: repo only | C: path-scoped | D: layered (selective) | E: layered (eager) |
|-----------|-----------------|--------------|----------------|------------------------|---------------------|
| Spec compliance | High | Medium (pre-1.0) | Medium | High | Medium (pre-1.0 LLMS.md) |
| Web-search reach | Yes | No | Yes (over-served) | Yes | Yes |
| Code-agent reach | Via README only | Yes | Via README only | Via README (high quality already) | Yes |
| Maintenance cost | 1 file | 60+ files | 60+ files | 2 files | 60+ files |
| Per-launch ceremony | +1 line append | +1 file commit | +1 file commit | +1 line append | +1 file commit |
| Reversibility | High | Low | Medium | High | Low |
| Aligns with `feedback_no_public_or_tag_without_explicit_yes` | Yes | Adds gate per launch | Adds gate per launch | Yes | Adds gate per launch |
| Honors [RES-018] hurdle for repo-level | N/A | Fails (no second-consumer evidence) | N/A | Passes (deferred) | Fails |

### Constraints

- `feedback_no_public_or_tag_without_explicit_yes` — every actual file ship
  to a public surface needs explicit go-ahead; the strategy here is
  authoring-only until launch.
- `feedback_readme_evaluator_audience` — READMEs already carry the
  evaluator-facing context burden; LLMS.md would either duplicate or
  diverge, both bad.
- `feedback_blog_publish_two_steps` precedent: routine repo work decoupled
  from public-site push; mirror this — author the website llms.txt in the
  swift-institute.org repo as routine work, push to the live site as a
  separately-authorized action.

### Per-repo flag for FUTURE consideration (not committed)

When the ecosystem revisits LLMS.md adoption, two sequencing options:

- **Cohort-launched packages** add LLMS.md at the same time, atomic across
  the cohort, to avoid version skew.
- **Individual launches** add LLMS.md as part of release-readiness. This
  would update the **release-readiness** skill with an `[RELEASE-AI-DISC]`
  rule.

Defer both until the convention triggers the threshold below.

## Outcome

**Status**: RECOMMENDATION

**Recommendation**: Option D (layered, selective).

**Concrete actions** (none authorized yet — awaiting per-action go-ahead per
`feedback_no_public_or_tag_without_explicit_yes`):

1. **Author** `swift-institute.org/llms.txt` and
   `swift-institute.org/llms-full.txt` in the swift-institute.org repo.
   Initial content scoped to current public surface (whichever packages
   are public when authored). Routine repo work.
2. **Push to live** swift-institute.org as an explicitly-authorized
   action when the user confirms.
3. **Defer** per-package LLMS.md until either:
   (a) LLMS.md spec reaches 1.0, OR
   (b) 2+ major code-agent vendors (Copilot, Cursor, Claude Code, Gemini CLI)
       document fetching it.
4. **Update** the `release-readiness` skill — when a package launches, the
   release flow appends one line to swift-institute.org/llms.txt pointing
   to the new package, and one paragraph to llms-full.txt. Add as
   `[RELEASE-AI-DISC]` rule. Routine, no per-action gate beyond the
   existing site-push gate.
5. **No org-tier action** beyond a one-line `.github/profile/README.md` for
   the top-level swift-institute org pointing to the website (already
   recommended practice; not load-bearing for AI discoverability).

**Re-evaluation trigger**: 6 months from this doc's date (≈ 2026-11-01) OR
on a confirmed sighting of either:
- An AI vendor publicly documenting llms.txt fetching as a retrieval
  feature.
- LLMS.md spec graduating from draft v0.1.

Whichever fires first, this doc is revisited and the deferred per-repo
question is reopened.

**What this doc does NOT decide**:
- The exact prose of the website llms.txt content (drafting is a separate
  authoring task).
- Which packages appear in the initial ecosystem index (driven by what is
  public at authoring time).
- Schema.org / structured-data markup on DocC pages (orthogonal lever, not
  in scope here).
