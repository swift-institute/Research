# Skill-verification taxonomy tier-1 extension — parallel subagent dispatch and three new resistant patterns

## What Worked and What Didn't

**Worked**:

The parallel subagent pattern was exceptionally productive for this task class. Four clusters returned ready-to-paste markdown sections in ≈4 min wall-clock. Three of four subagents caught their own count-tally discrepancies during the per-skill distribution step and self-corrected — a quality artifact of (a) the explicit instruction *"All N requirement IDs walked. Do not skip rules"*, (b) the rigorous table format that exposes mismatched counts to the agent's own verification step, and (c) the request for a per-skill distribution row that forces enumeration. The format-instrument-first dispatch (each subagent reads the pilot doc to internalize column headers, bold-lowercase class formatting, and file:line discipline before opening the assigned SKILL.md) eliminated post-aggregation reformatting cost. Confidence in subagent outputs was high enough that I lightly edited only γ-mark over-eagerness (one DOC-026 → γ-2 mark dropped during composition) and self-corrected count tallies.

The line-count-balanced cluster sizing (1420 / 1508 / 1533 / 1636 lines across 4 / 3 / 3 / 3 skills) produced approximate wall-clock parity. Skill-count balancing (3-3-3-4) would have over-loaded the 4-skill cluster relative to a 3-skill cluster containing existing-infrastructure (1203 lines on its own).

The carry-forward-pilot-Part-1 discipline saved real time and tokens: I did not re-derive the three-class-vs-four-class reconciliation and instead cited the pilot's verdict in §1 of the extension doc. This kept the extension doc focused on tier-1-specific signal (per-skill classification, aggregate distribution, new resistant patterns) rather than re-litigating the foundation.

**Didn't work cleanly**:

Subagent γ-mark over-eagerness produced occasional wrong cross-references — DOC-026 ("Flat catalogue layout") was annotated as `→ γ-2` but γ-2 is YAML lint + broken-symlink, not catalogue-subdirectory check. I caught this during composition but it's evidence that subagents apply the cross-reference rubric loosely when the γ category description is ambiguous (γ-2 reads as "mechanical hygiene" in some prose, narrower as "YAML lint + symlink" in §3.4.5). Future dispatches should give subagents a tighter γ-class description with worked examples.

One subagent (Cluster A, documentation skill) read 7 sibling files (`inline.md`, `catalogue.md`, `topical.md`, `tutorial.md`, `landing.md`, `style.md`, `visual.md`) in addition to `documentation/SKILL.md`. The file:line cites were correctly against `documentation/SKILL.md` (which indexes the requirement IDs declared in the sibling files), but the deeper investigation was beyond the strict assignment. Outputs were good — arguably better than a SKILL.md-only read would have produced — but the scope expansion should have been surfaced explicitly. The assignment text said "walk every `[REQ-ID]` in each SKILL.md" and the subagent interpreted this as "walk every ID indexed by SKILL.md, including those defined in sibling files."

The subagents' first-pass count tally errors (3 of 4) initially caused friction at aggregation time: I had to read each cluster's output for re-tally markers and use the corrected versions. Net outcome was correct, but the noise indicates the distribution-table-shape would benefit from a single-pass discipline (count first by going row-by-row, then write).

**Confidence-was-low moments**:

Confidence dropped at the commit-shape decision. The pre-existing dirty working tree had three sessions' worth of pending edits to `_index.json` (wasm-ci entry + pilot entry from prior sessions, plus my extension entry). The new `feedback_triage_dirty_worktree.md` rule (added mid-session per the MEMORY.md modification reminder) explicitly forbids blanket `git add -A` when anything is outside the task; the file-level analog of that — "what does it mean to commit `_index.json` when it carries prior-session edits?" — has no codified disposition rule. I made the conservative call (commit only the new doc; leave shared-file edits in working tree; surface to user) but the call was non-obvious and would benefit from an explicit reflect-session rule.

Confidence dropped again at the handoff-stamping step. ## Patterns and Root Causes

**The parallel-subagent-dispatch pattern is reusable beyond classification tasks.** Three pre-conditions made it work here: (a) the deliverable was a structured artifact (per-skill table + distribution) that aggregates by concatenation; (b) the format instrument (pilot doc) was a self-contained reference each subagent could read independently; (c) the assignment was bounded — exactly N skills per cluster, with explicit instruction to "not skip rules." When these three pre-conditions hold, parallel subagent dispatch with line-count-balanced clusters + format-strict prompts + structured tables produces near-linear speedup with high accuracy. The same shape applies to: cross-package conformance audits ("verify every L1 type's `~Copyable` discipline"), ecosystem-wide dep-direction checks ("for each L3 package, verify deps are L1-only"), and skill-corpus consolidations ("classify every requirement against this rubric"). Tier-2 and tier-3 dispatches in the current sweep should reuse the cluster shape directly with adjusted line-count balance.

**The "tier predicts density" assumption fails on rule-shape variance.** The handoff brief framed tier-1 as "the cluster expected to have the highest mechanical-rule density." This was largely true (57% mechanical, +12 pp over pilot) but with three counterexamples that were structurally tier-1 by scope yet rule-shape-non-mechanical: ecosystem-data-structures (91% semantic — pure catalog selection), swift-institute-core (0 IDs — manifest), swift-institute (1 ID — degenerate single-rule). The pattern is that **tier classifies leverage and scope, not rule shape**. Tier-2 (code-shape skills) is likely *more* mechanical-heavy than tier-1 in absolute terms because code-shape rules are AST-checkable by definition; tier-3 (voice/process) is likely less mechanical. But within any tier, density variance can be wide — the handoff's "tier-1 is mechanical-heavy" framing should not become a self-fulfilling expectation that tier-2/3 dispatches calibrate against.

**Resistant patterns scale up at scale.** Pilot's three resistant patterns (composite / workflow / external) each grew at tier-1 in absolute count (5 → 12, 5 → 14, 3 → 12) but the resistant-set fraction shrank slightly (22.8% → 19.9%). Three new patterns surfaced that the pilot did not predict, and the cause is structural: the pilot's three pilot subjects (code-surface, primitives, blog-process) were drawn from across the corpus to span the three classes; tier-1 is a thematic cluster (infra/process) where rules concentrate around catalog-style decision-trees (existing-infrastructure), platform-API gaps (github-repository), and operational illustrations (package-export). The new patterns reflect those structural concentrations, not weaknesses in the three-class taxonomy itself. The taxonomy remains structurally sound; the annotation surface needs to grow with each tier's structural concentration.

**Dirty-working-tree disposition needs codification.** The session's commit-shape decision was non-obvious because the new `feedback_triage_dirty_worktree.md` rule covers the staging-decision dimension ("don't blanket `git add -A`") but does not address the file-level dimension ("when a shared file has prior-session edits + my edit, do I commit it?"). The conservative path (commit only my new files, leave shared-file edits in working tree, surface in verdict) preserves session-attribution clarity but leaves the index in a temporarily-broken state (entry added in working tree but not in HEAD). The opposite path (commit the shared file with all pending edits) conflates sessions but produces consistent state. Neither is universally correct; the choice depends on coupling. For my session, the pilot doc was tightly coupled (extension cites pilot 30+ times) but `wasm-ci-strategy-and-sdk-toolchain-coupling.md` was not coupled; the JSON entries for both prior sessions were already-staged-but-uncommitted; cleanly extracting *only* my edit from `_index.json` would have required JSON manipulation that risks breaking prior session's work. Codifying this disposition as a `[REFL-014]` rule would close the gap.

**Branching handoffs need explicit stamp-location guidance.** [HANDOFF-010] step 5 reads "stamp a verification status line in HANDOFF.md" — the literal text refers to sequential `HANDOFF.md`, but the verification-stamp protocol applies equally to branching handoffs (`HANDOFF-{topic}.md` per [HANDOFF-005]). Stamping a branching handoff at workspace root is correct (the handoff file IS the disposition record); future-session agents resolving the apparent rule-text-vs-intent gap should not have to re-derive this. The skill-text fix is one-line.
