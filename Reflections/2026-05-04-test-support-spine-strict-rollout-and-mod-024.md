## What Worked and What Didn't

**What worked**:

- **Pilot-first → bulk-script** sequence. Manual pilot on Comparison Primitives validated the pattern; the script then handled 47 packages with only two surgical fixups (witness, base62). Net: 1 manual + 1 script + 2 fixups vs 48 manual.
- **Two-pass anchor algorithm**. Pass 1 (existing-TS priority) covered most packages; Pass 2 (will-be-created in this pass) cleanly handled the algebra family's chain (Magma → Monoid → Semilattice/Semiring → Ring → Field → Module). The chain inversion arrived at correct anchors deterministically.
- **`preflight-test-support-spine.py` as the spec**. Re-running the audit post-rollout (`Violations: 0 | Missing: 0`) gave decisive go/no-go feedback. The script is also what Phase β's CI gate will run, so the same artifact is both verification tool and enforcement mechanism.
- **Stash-and-rebuild verification** for false-positive triage. When 3 packages failed the post-rollout sweep (layout, string, symmetry), I stashed the TS-spine changes and rebuilt to confirm the failures pre-existed — saving the principal from chasing imaginary regressions.
- **Clean-build reveals masking**. The user's "rm -rf .build / clean build" intervention on string-primitives showed it was a stale-cache false negative; the same intervention on cache-primitives and parser-primitives surfaced the pre-existing `swift-array-primitives` source rot that had been masked by stale cache for those packages.

**What didn't work**:

- **Hand-curated `EXISTING_TS_PRIORITY` list**. I assembled the 27-entry priority list by intuition rather than deriving it from the audit JSON's `ts_targets` field. Two false positives (`swift-ascii-primitives`, `swift-dependency-primitives`) — both have no test targets and no TS targets, but I had assumed they did. The bug surfaced as broken anchor refs in `swift-witness-primitives` and `swift-base62-primitives`. The fix was easy; the lesson is that the audit JSON is the authoritative source and tooling should derive from it, not from the implementer's mental model.
- **Brief `.underlying` overcorrection on Symmetry**. After fixing `__unchecked:ordinal:` → `_unchecked:ordinal:`, I also changed `Int(ordinal.rawValue)` to `Int(ordinal.underlying)` on the assumption Ordinal was Tagged. It isn't — `Ordinal` is a regular struct with `public let rawValue: UInt`. Reverted within seconds. Pattern: fix the actual reported error first, don't pre-emptively apply adjacent renames without verifying the type structure.
- **Stale `.build` masked pre-existing rot for several packages** in the initial sweep. parser-primitives passed first; failed after clean rebuild. Without the user's clean-build directive I would have reported "all clean" while harboring three latent failures.
- **Test-target-name vs directory-name drift** caught me unprepared. format-primitives's "Formatting Primitives Tests" testTarget name with "Format Primitives Tests" directory worked under SPM auto-detection only because there was no other test directory at `Tests/`. Adding `Tests/Support/` broke the auto-detection. Pre-existing inconsistency, surfaced by adjacent change.
- **The compacted summary's premise on "211 missing"** was inflated by stale package counts (swift-foundations 139→72 actual). Caught early in the session by the resuming-agent verification; recorded in the handoff. Premise staleness is a real cost of long multi-session arcs.

## Patterns and Root Causes

**Hand-curated capability lists are bug magnets when the authoritative source is mechanical.** The `EXISTING_TS_PRIORITY` list was 27 entries I curated by skimming an earlier grep output. The audit JSON's `ts_targets` field is the authoritative source — derived mechanically from `Package.swift` parsing. My hand-curation introduced 2 false positives (~7% defect rate) that propagated into broken anchor refs in 2 downstream packages. The fix in this case is one Python query (`have_ts = [p['package'] for p in prim['packages'] if p['ts_targets']]`); generalized, the lesson is: when tooling needs a list of "packages with capability X" and an audit script already classifies them mechanically, the tooling MUST derive from the audit's output, not from a hand-curated list. This generalizes to CI workflow logic (which packages have private deps, which have embedded variants, which have benchmarks), to release tooling, and to docs generation. The hand-curated form is faster to write but slower to be correct and harder to keep correct as the ecosystem evolves.

**Strict-uniformity over empirical-minimum is a contributor-cost decision.** The audit empirically said 16 packages need shells under the pragmatic disposition (transitive-closure–reachable from Tagged TS); strict says all 117 packages with tests get a shell. My recommendation was pragmatic on the basis of "fewer empty files." The user chose strict on the basis of "contributors learn one rule everywhere." That trade-off is worth naming as a class: when the spec lives in human heads (contributors landing PRs months from now, no longer the principal who designed it), uniform-everywhere has a structural advantage over minimal-where-needed even when the minimal version is empirically tighter. The cost of one empty file per package is bounded; the cost of "did this package need a shell or not? let me re-derive the rule from scratch" recurs every time a contributor writes a new test file. Strict-mode pays the one-time scaffolding cost to permanently retire that re-derivation cost. This is the same shape as [MOD-005] (umbrella re-export) chosen over per-target-import — uniformity over minimalism for human-facing surfaces.

**Bulk verification surfaces ecosystem rot as a side effect.** The Phase 2a sweep (build-tests across all 48 modified packages serially) was framed as "verify my changes." The actual yield was a snapshot of ecosystem build health: `swift-array-primitives` main module has pre-existing `~Copyable` source errors that block downstream `swift-cache-primitives` and `swift-parser-primitives` test builds; `swift-symmetry-primitives` test target has type-inference timeouts in `Rotation Tests.swift`. None of those are TS-spine-related. They surface here because (a) the TS-spine work touches every package, exercising the dep chain end-to-end, and (b) per-package builds previously hadn't been run with a clean .build often enough to expose the rot. The pattern: ecosystem-wide structural changes carry the side effect of ecosystem build-health snapshots — and that side-effect data is high-value for sequencing follow-on phases (Phase 2b/c/d should land array-primitives source fix BEFORE expanding the spine to standards/foundations to avoid repeating this discovery).

**Test-target-name vs directory-name drift is a latent SPM auto-detection trap.** SPM's testTarget path auto-detection works as long as one and only one directory at `Tests/` matches the testTarget name pattern. format/position primitives had a name-vs-directory mismatch ("Formatting Primitives Tests" vs `Format Primitives Tests/`) that worked because no other test directory existed. Adding `Tests/Support/` broke the auto-detection — the testTarget started claiming both directories. The fix is one line of explicit `path:`. The latency: this inconsistency has lived in those packages since they were written, but no one noticed until adjacent state changed. Pattern: adjacent changes can surface latent inconsistencies without "breaking" anything you actually changed. The reflex "my change introduced this" is wrong; the reality is "my change exposed this." Catching the difference is the cheap part; surfacing-for-clarification before fixing is the discipline.
