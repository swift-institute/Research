# SPM-stall: comprehensive global mirror as the structural mitigation

## What Happened

The session began with a branching handoff (`HANDOFF-spm-stall-duplicate-clock-primitives.md`) describing a deterministic SwiftPM stall at swift-foundations/swift-io and swift-foundations/swift-executors: `swift-build` spinning at ~99% CPU, no swift-frontend children, no `.build/` artifacts, the planner livelocked after `swift package resolve` completed. The handoff's leading hypothesis was a workspace-topology defect (duplicate `Clock Primitives` product references in two `Package.swift` files) and the brief authorized commit + push if that proved to be the root cause.

Phase 1 — diagnosis. Reproduced the stall (clean state, T+1:23 elapsed at 99% CPU, 0 `.o` files, 0 swift-frontend children). Confirmed the duplicate-product entries by mechanical scan: `swift-foundations/swift-kernel/Package.swift:174-175` (Kernel Clock target) and `swift-iso/swift-iso-9945/Package.swift:290-291` (ISO 9945 Kernel Clock target), each had two byte-identical `.product(name: "Clock Primitives", ...)` lines. Removed both duplicates (commits `6073f02` and `0fbdb32`); confirmed warnings disappear post-fix; **stall persists identically**. Hypothesis (a) refuted as the cause of the stall.

Phase 2 — exhaustion of workspace-side mitigations. The user prompted with "did you already try removing .build?" — the cheap workaround. Tried `swift package purge-cache && swift package reset` (cleared 400MB manifest cache + 254MB repos cache); stall persisted. Web search surfaced `swiftlang/swift-package-manager#9441` and `PR #9493` (Linux/KVM thread-pool starvation, merged 2025-12-12) as the closest known issue. Tried Swift 6.4-dev (DEVELOPMENT-SNAPSHOT-2026-03-16-a, post-PR-#9493): same `_platform_memmove`-deep planner hot loop. PR #9493 ruled out as the same bug. Tried single-target build (`swift build --target "IO Events"`): stalls identically. Tried commenting out swift-witnesses (the macro package): partially inconclusive — stall persisted but 5 other macro-using foundation packages keep swift-syntax in the closure regardless. Tried per-package mirror redirecting swift-syntax to local file:// path: stall persisted. Tried global mirror (single URL): stall persisted.

Wrote a supervisor report concluding Path B (file upstream issue against `swiftlang/swift-package-manager`). User redirected: **"we want to get a structural solution in place. It's unacceptable that just having a few packages remote is causing all this issues."** When I clarified that the bug isn't actually about remote fetches per se (mirror redirect to local file:// also stalls), the user clarified the architecture: "I want URLs in Package.swift AND it resolve to the local path package, for ease of development. eliminate macros from swift-io's tree is a red herring."

Phase 3 — comprehensive mirror. Inventoried URL deps in our owned workspace: 5 total (4 user-owned: `swift-carrier-primitives`, `swift-tagged-primitives`, `swift-ownership-primitives`, `swift-property-primitives` + Apple's `swift-syntax`). The previous mirror experiment (Phase 2 above) had covered only swift-syntax. Built `~/Library/org.swift.swiftpm/configuration/mirrors.json` mapping all 5 URLs → local file:// checkouts. Cleaned state, ran `swift build` at swift-io: planner advanced through resolution, manifest evaluation, build planning, and INTO COMPILATION — generated 2685 `.o` files, 303 `.swiftmodule` files, surfaced real source-level compile errors in `swift-io/Sources/IO Events/Event.Actor.swift` (Kernel.platform / Kernel.POSIX ambiguities). **Stall completely resolved.**

Phase 4 — productionize. Scaffolded `coenttb/swift-package-mirrors` (private repo per user direction): templated `mirrors.template.json` with `__WORKSPACE_ROOT__` placeholder, `Scripts/install.sh` (substitutes workspace root + writes to `~/Library/org.swift.swiftpm/configuration/`), `Scripts/verify.sh` (asserts every local path exists), `.github/workflows/verify.yml` (JSON syntax + placeholder + shellcheck + dry-run install on Linux), README. Tested install + verify end-to-end against `/Users/coen/Developer`; install.sh produces byte-identical mirrors.json to the manually-written one that proved the fix. Committed initial commit `82ef956` locally; created GitHub remote (private) + pushed via `gh repo create coenttb/swift-package-mirrors --private --source=. --remote=origin --push`. Updated `swift-institute/Research/spm-stall-investigation-2026-05-01.md` with TL;DR rewrite (`6be4136`) and `swift-institute/Audits/post-path-x-architecture-review-2026-04-30.md` SPM-stall section (`235c2ed`); pushed both.

Wave 4a Sites 1+2+3 closed: 5-of-7 consumers green pre-mitigation; comprehensive mirror unblocks the remaining 2; remaining source-level errors in swift-io are a separate concern deferred to another agent per user direction.

## What Worked and What Didn't

**Worked**:
- The user's two redirects ("did you try removing .build?" and the structural-solution directive) corrected premature conclusions about workspace-side mitigations being exhausted. The investigation had run out of energy at the wrong point twice — at the duplicate-product fix (where I'd called it "the fix is real but doesn't resolve the stall") and at Path B (where I'd recommended upstream filing). Both times, the next step was actually within reach.
- Web search ([ISSUE-001] dev-toolchain check + closest-existing-issues search) was high-leverage — surfaced #9441 and PR #9493 quickly; ruled out PR #9493 quickly via Swift 6.4-dev test.
- The comprehensive-mirror approach was cheap to test once identified (~5 minutes to write the JSON + run the build) and the empirical evidence was immediate (2685 `.o` files generated within 2 minutes).
- The repo scaffold worked first-try: install.sh + verify.sh end-to-end test on the same workspace produced byte-identical output to the manually-written mirrors.json.
- The hygiene-fix commits (duplicate Clock Primitives removal) landed cleanly and are independently useful even though they didn't fix the stall.

**Didn't work**:
- Phase 3's single-URL mirror experiment generalized too aggressively. Mirroring only swift-syntax produced a "mirror doesn't help" conclusion that wasn't quite right — the correct conclusion was "1-of-5 URL coverage doesn't help; comprehensive coverage not yet tested." The user spotted the gap immediately when I asked "structural how?" — they listed all 4 user-owned URLs that needed mirroring, including ownership-primitives and property-primitives which I hadn't enumerated.
- I autonomously concluded "Path B" (file upstream and wait) when there were still workspace-side options to try. The supervisor's dispatch to file upstream was already prepared before the user redirected. If the user hadn't intervened, I would have filed an upstream issue prematurely.
- I missed `swift package --help` triggering the same stall pattern — discovered accidentally when a stale process held a `.build` lock for 18 minutes. That's a sharp signal (the bug fires on manifest load alone, not just `swift build` planning) but I only noticed it when it became an obstacle, not as a deliberate diagnostic step.
- I didn't grep the workspace's URL deps comprehensively at Phase 3.1. The brief mentioned swift-syntax + swift-carrier-primitives URL but I only set up the mirror for swift-syntax. A complete URL inventory at Phase 3.1 would have led directly to the comprehensive-mirror approach instead of through a misleading single-URL false negative.

**Confidence assessment**: highest confidence at Phase 1 (mechanical reproduction, byte-level evidence). Lowest confidence at Phase 3 — the single-URL mirror experiment's negative result felt definitive to me but was actually an under-specified experiment. The user's redirect was the corrective signal.

## Patterns and Root Causes

**Pattern 1 — negative-experimental-conclusion-without-coverage-scope.** The Phase 3 mistake: tested mirror with N=1 URL covered, concluded "mirror doesn't help in general." The proper conclusion was "mirror with 1-of-5 URL coverage didn't help; full coverage not yet tested." The two are categorically different. The premature generalization sent the investigation toward upstream filing when comprehensive coverage was the actual fix. This is a recurring failure mode whenever an experiment's coverage scope is implicit rather than stated. The corrective discipline: when reporting a negative experimental result, the conclusion sentence MUST cite the experiment's coverage scope alongside the result.

The user pointed this out indirectly: when I described the Phase 3 result as conclusive, they asked the right meta-question ("can't we have a global mirror?") — pushing past the experiment's specific failure to the broader question of whether the approach itself was viable. The discipline I'd missed: state coverage explicitly so the gap between "this experiment failed" and "this approach is exhausted" is visible.

**Pattern 2 — handoff completion + user redirect produces premature dispatch closure.** The supervisor dispatched a Path B sequence (draft upstream issue → halt for review → file). I had drafted the issue body and was about to surface for principal review when the user redirected. The Path B dispatch felt complete from inside its own framing — "we exhausted workspace-side mitigations, surface upstream, halt for review" — but from outside, "exhausted" was a function of the experiment-coverage gap (Pattern 1). The supervisor's role and the user's role aren't redundant: the supervisor optimizes for sound process within a chosen direction; the user can change directions. Treating the user's redirect as authoritative-over-supervisor was correct (per `feedback_user_intent_over_principal_tangents.md`), but I could have surfaced the directional question myself before drafting the upstream issue body — "given comprehensive mirror is unverified, do we want to try that before filing upstream?"

**Pattern 3 — Swift 6.3 / 6.4-dev planner stall is ecosystem-shape-dependent.** The stall isn't about the absolute workspace size (94 packages) or any single package. It's about the URL/local identity-dedup walks SwiftPM performs when both forms exist for the same package. Each URL/local pair adds a walk; at 5 pairs (or some threshold ≤5), the planner livelocks. Mitigation by removing one URL/local pair fails because the others still trigger. Mitigation by removing all URL/local pairs (via comprehensive mirror redirecting URL → local) succeeds because the dedup walks have nothing to dedup. This generalizes: any workspace topology with multiple URL/local pairs is potentially affected. The `coenttb/swift-package-mirrors` recipe is the workspace-side mitigation; the upstream SwiftPM bug remains unfiled (deferred to user disposition).

**Pattern 4 — auto-mode boundary management.** The user toggled auto-mode off mid-session ("we want a structural solution") to engage directly. After the structural fix landed and committed, auto-mode came back. The transitions were the right shape — auto-mode for execution, direct engagement for direction-setting. The skill-side lesson: when in auto-mode and the conclusion of a deep investigation is "let's file upstream," that's the natural turn to surface for direction-setting rather than autonomously proceeding (per CLAUDE.md "exploratory questions get 2-3-sentence recommendations, not decided plans"). I had drafted the supervisor report rather than asking — a small but real overreach.
