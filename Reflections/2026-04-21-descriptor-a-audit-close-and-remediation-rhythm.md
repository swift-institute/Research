## What Worked and What Didn't

**Worked.** The A+audit exit path converted a stuck, cognitively-taxing architectural debate into ~30 minutes of mechanical git work across five repos plus an archive commit. Six-iteration Descriptor design preserved losslessly on `descriptor-migration-wip` branches; main branches back to known-green baselines; audit published as a real deliverable rather than an intermediate artifact. Nothing lost, nothing rushed into a wrong decision. The "branch WIP first, then revert main" sequence made the revert non-destructive — no `reset --hard`, no force-pushes, just `git checkout -b → git add -A && git commit → git checkout main`, which works cleanly when all phase work is uncommitted (as it was).

The remediation cycle's one-commit-per-finding + audit-tracker-update-per-commit rhythm held for nine fixes straight without rescoping. Pre-stated "if N files outside target, halt and re-scope" budget never fired. Each fix stayed under ~30 minutes including discovery + build + commit + tracker update. The audit document evolved from static findings-table into a living tracker via three progressive in-place edits (Progress at a Glance, then RESOLVED rows as they landed). A future session picks up from the tracker state, not from a conversation summary.

Supervisor-critique role under user direction was load-bearing. User asked `"give your take on this supervision"` multiple times at decision points, which surfaced asymmetric option framing and scope-escalation drift the parent supervisor had not self-diagnosed. The critique happened inside the session rather than in a separate retrospective, which meant the user could act on the critique immediately (e.g., the abolition-scope-too-wide observation led user to stop short of authorizing investigation).

**Didn't.** Arc 1 stamped HANDOFF.md's verification checklist per [SUPER-011] before Arc 3 archived the file, so technically the ground-rules obligation was closed. ## Patterns and Root Causes

**Pattern: audit-vs-remediation separation as mechanical, not aspirational, discipline.** The parent session's Descriptor tangle happened because one audit finding's remediation ballooned into six design iterations — the `/platform` audit was treated as input to a continuous work-stream rather than a discrete deliverable. This session's nine-fix remediation rhythm succeeded because the audit document itself held the line: finding enumerated → commit cites finding ID → tracker row appended → next finding. The audit isn't a suggestion deck; it's a tracker. The separation becomes mechanical when the tracker-update-per-commit step is not optional. Root cause of the parent tangle: no tracker existed, so each remediation attempt had to re-derive scope from the findings list; without a tracker, "is this finding handled?" becomes ambiguous at every session boundary, and the remediation defaults to "keep iterating until it feels done" — which for a cascading-coupling finding never converges. With a tracker, "this commit closes P1.7a" is unambiguous; the next session opens on P1.7b, not on re-debating P1.7a. The rule generalizes: any multi-finding audit needs a tracker table with per-finding status column, updated per commit, or the remediation drifts into re-planning.

**Pattern: audit "mechanical" effort classifications silently encode optimism about cross-type bridging.** P1.6 Mode A was labeled mechanical. The Random delegations needed error-type catch-and-map bridges because Random_Primitives.Random.Error and Kernel_Random_Primitives.Kernel.Random.Error happen to be distinct types that collided at the delegation boundary. The audit's effort/classification column reads as "how much code?" but the hidden axis is "how many types need to bridge across the change?" — a mechanical substitution of `arc4random_buf(...)` → `Darwin.Kernel.Random.arc4random(...)` has zero bridging; a type-preserving throws contract across two error types has one bridge per site. The audit's effort estimate was correct as a LOC count; the classification was wrong as a risk label. The fix is a column addition, not a rewrite. Provenance ties to a broader pattern observed in P0.1 as well: each Descriptor-migration iteration discovered coupling the audit had not encoded (Event.Driver holds Descriptor; Event.ID holds Descriptor bridge inits; Interest typealiases to Descriptor.Interest; L1 executor-primitives holds Event.Source). The audit's finding table enumerated leaf violations; it did not enumerate the coupling graph between them. A coupling-graph column at audit-write time would surface these earlier.

**Pattern: tentative language from a fatigued principal is not firm approval.** User said `"Kernel.Event.Source might have to move to L3"` — the parent supervisor treated this as soft approval and proceeded to concretize the move. When I reviewed, I flagged the tentativeness. Later, user said `"we should really look harder at whether kernel-primitives should be abolished entirely"` — parent supervisor escalated this into a multi-session abolition investigation scope. Same failure mode both times: tentative conditional ("might have to", "should really look harder") read as firm direction when it was exploratory language. Root cause: principal-agent interaction under principal fatigue conflates option-generation with option-selection. Fatigued principal thinks out loud; agent treats the thinking-out-loud as dispatch. The symmetric observation on the subordinate side: subordinate under fatigue may take tentative direction as dispatch because the cognitive cost of asking for firm confirmation feels higher than the cost of executing. Both are shortcuts that produce wrong work. The `/supervise` skill has escalation protocols for structural blockers but no explicit rule about tentative-language detection at the principal-input side. The rule: when principal uses conditional or exploratory language ("might", "maybe", "should probably", "worth looking at") in response to a scope-escalation question, the subordinate-or-supervisor explicitly surfaces the tentativeness and asks for firm yes/no before scoping new work. Concrete test: rewrite the principal's sentence with `"I want you to"` prefix — if it reads false, the original was exploratory, not directive.
