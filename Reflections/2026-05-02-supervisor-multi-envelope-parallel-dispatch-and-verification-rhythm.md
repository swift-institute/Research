## Patterns and Root Causes

**The research-then-dispatch protocol surfaces structural conflicts that direct dispatch hides.** Two of two research-then-dispatch cycles this session surfaced structural surprises (Item 1.5 Phase 2 BLOCKER + Item 3.5 premise inversion). In each case the surprise would have caused wasted code work if dispatched directly: Item 1.5 would have produced an uncompileable L1 redesign; Item 3.5 would have moved files in the wrong direction. The protocol's value is exactly this surfacing function — a 2/2 hit rate in one session is a strong empirical signal that "the protocol IS the value, not a procedural overhead." Subordinate A's reflection adds the corollary: research-doc deliverables for L1 redesigns should verify Package.swift import constraints alongside type-system shape (three-axis verification: type-system + Package.swift deps + consumer cascade). The supervisor-side mirror is: when dispatching a redesign that's grounded in a research doc, verify the research doc's layering claims against current Package.swift state before authorizing.

**Verification-via-empirical-evidence is supervisor work, not subordinate trust.** [SUPER-009] says verify against principal-authored criteria, not subordinate self-report. This session executed that consistently — every close-report had cited empirical evidence (git log / grep / file read) and supervisor verified those citations before disposition. The discipline produces zero false-positives in this session's data, but the discipline matters precisely because the failure mode (accepting subordinate attestation when reality differs) is invisible until it costs a regression. This is structurally similar to the [REFL-006] re-verify-after-edit rule — both close blind spots that would otherwise stay closed by accident.

**Parallel-dispatch coordination via explicit MUST NOT clauses scales linearly.** Edit-zone non-overlap across two subordinates required explicitly enumerating the other subordinate's edit zones in each dispatch's MUST NOT clauses. Both subordinates assimilated the discipline well enough to self-classify cross-package diffs as "the other subordinate's work" without supervisor intervention. The pattern would extend to N>2 subordinates with similar discipline (each dispatch enumerates ALL other in-flight dispatches' edit zones), but coordination overhead grows quadratically with N. For N=2 the overhead was negligible.

**Reframe-not-defend when premise inversion surfaces.** When subordinate pre-flight surfaces a stale-framing dispatch, the supervisor's natural temptation is to defend the original framing (sunk-cost on the dispatch). The principled response is to confirm the reframe immediately, document it in the research/audit doc as load-bearing artifact, and re-dispose. This session executed this twice (Item 1.5 + Item 3.5) without defensiveness. The supervisor pattern: dispatches are hypotheses; subordinate empirical work tests them; reframes are valuable signal, not friction.

**Acknowledge subordinate course-corrections explicitly to keep the channel open.** When Subordinate A chose `package` over `internal`, the choice was better than my dispatch suggested. Acknowledging this explicitly ("smart choice — better than my originally-suggested `internal`") signals that the supervisor expects + values such corrections. The opposite signal (silent acceptance or worse, ignoring the deviation) would close the channel for future course-corrections. This session's high parallel-dispatch throughput depended on subordinates feeling empowered to deviate when they had better information.

**The "make progress" + "first principles correct" tension resolves via empirical scoping.** Auto-mode + parallel dispatch creates pressure to dispatch quickly. First-principles-correctness creates pressure to research before dispatching. The resolution this session: dispatch empirical scoping cycles when stakes are non-trivial (Item 1.5 Phase A experiment + Phase B Path β scoping; Item 3.5 Phase 0 research-doc kickoff). The empirical scoping cycles are themselves "make progress" work — they're not deferred research, they're load-bearing dispatches. The pattern collapses the false dichotomy.

**Cross-workstream coordination is fundamentally a supervisor-only view.** Neither Subordinate A nor Subordinate B can see the other's session. Patterns that emerge across both workstreams (the 2/2 research-then-dispatch hit rate; the parallel-dispatch coordination discipline; the reframe-not-defend protocol) are visible only at the supervisor level. The reflection ordering you proposed (subordinates first, supervisor last) makes structural sense — each layer captures patterns visible only at that layer.
