## Patterns and Root Causes

**Pattern 1: typealias-chain conflict invisibility**. When package A declares `public typealias X = OtherModule.Y`, any extension `extension A.X.Z { ... }` becomes `extension OtherModule.Y.Z { ... }` at compile time — adding to OtherModule's namespace. This is invisible at the source level: an iso-9945 source author writing `extension ISO_9945.Kernel.Descriptor` *looks like* they're authoring iso-9945-resident code, but if `ISO_9945.Kernel = L1.Kernel`, they're actually adding to L1. For a NEW declaration, this means conflicts with the foreign module's existing types are silent until the compiler resolves them. The platform skill describes the typealias mechanism for cross-platform unification ([PLAT-ARCH-005] L3-typealias pattern) but doesn't codify the *conflict mode*: that adding new types via a typealiased namespace path conflicts with the underlying module's existing types at the same path. The Phase 1.5 attempted relocation made `POSIX.Kernel.Descriptor` (top-level POSIX, not aliased) work because `POSIX` was a fresh enum at iso-9945; the namespace correction's literal interpretation `ISO_9945.Kernel.Descriptor` would have failed because that path goes through the typealias to L1.

**Pattern 2: principal-directive vs structural-constraint reconciliation**. Three different principal-directives in this session pointed in three different directions: (a) the original dispatch text "POSIX.Kernel.Descriptor at L2 in iso-9945"; (b) the namespace-ownership note "only swift-posix declares POSIX, use ISO_9945-prefixed"; (c) the implicit structural constraint "L1 Kernel.Descriptor still exists, don't touch L1." Each was internally consistent; pairwise, any two were satisfiable; all three together were not. The subordinate's job in this case is to surface the trilemma rather than pick one. I picked (a) initially (executed Phase 1.5 as dispatched), then the principal surfaced (b) post-execution, and I navigated to "defer entirely" without explicitly raising the trilemma. A better path: when the second directive arrives, the response is "your two directives plus the structural constraint form a trilemma — here are the three options, which do you authorize?" Per [SUPER-005] this would be class (c) escalation (would change the user's stated goal — relocation goal becomes deferral goal).

**Pattern 3: Research-doc recommendations vs dispatch overrides**. The Research doc v1.0.0 recommended "status quo timing — defer to post-Cycle-23 cleanup" over "pivot now" with explicit reasoning. The principal's HANDOFF.md dispatch overrode to pivot-now. This is a permitted move (the principal authorizes; the doc recommends). But the pivot's downside played out exactly as the doc predicted: ~30 commits of churn, supervisor-stamped clean, then reverted. Net: zero progress on type relocation; ~40 commits of churn (12 forward + 6 revert + 1 doc-cleanup); the typed-form value retained could have been achieved without the type-relocation attempt by simply using L1 `borrowing Kernel.Descriptor` from the start. The Research-doc-recommendation-bypass cost was paid in full. This generalizes: when a Research doc explicitly recommends X timing, and the dispatch overrides to Y, the supervisor block (or the dispatch itself) SHOULD include a `fact:` entry naming the doc's preferred timing as the prior-art reference, so the divergence is visible in real-time rather than only retrospectively.

**Pattern 4: subordinate-ground-rule-completeness illusion**. The Phase 1.5 supervisor block had nine well-typed entries; my Wave-by-wave verification was conscientious; the supervisor stamped clean. None of that prevented the namespace-ownership omission. The supervisor block is complete on its declared axes; it cannot be complete on undeclared axes. A subordinate who satisfies all entries cleanly is doing their job — but cleanliness on declared axes doesn't certify the dispatch is sound. This argues for *pre-dispatch sanity checks* that aren't axis-specific: mechanical "what could fail that isn't named here?" passes the principal SHOULD run before authorizing dispatch. Per [SUPER-018] (Supervisor Re-Reads Skill at Intervention Points), the analogue for dispatch is: re-read the relevant ecosystem-state checks (namespace ownership map, package-dep graph, layer-direction rules) at dispatch time. The 30-second namespace-ownership grep before authorization would have surfaced this.

**Root cause** (synthesizing 1–4): *the supervisor model's risk surface defaults to what the dispatch text names*. When the dispatch is wrong about the risk surface, the supervisor block can be perfectly executed and still produce a defective outcome. The Research doc recommended status-quo precisely because it judged this risk surface (Phase-1 momentum invalidation) higher than the principal did. The principal chose pivot-now; the dispatch text didn't enumerate "structural constraints not yet surfaced"; the supervisor block's nine entries faithfully reflected the dispatch text; supervision verified clean. The pivot was structurally unsound from the moment of authorization — the namespace-ownership constraint existed before Wave 0 — but no explicit check fired because no axis named it. This is a *pre-dispatch* gap, not a *during-execution* gap.
