## What Worked and What Didn't

**Worked.** The cluster-and-commit cadence kept history reviewable — each commit's message had verdict counts and the specific experiments it covered, so a reader can see at a glance what the session did. The `add-reval.sh` script, once the regex was widened, handled almost every file uniformly. Recognizing FIXED as a needed fourth verdict came from holding two pieces of evidence side by side (documented expected outcome vs observed clean build) rather than just classifying based on observed behavior alone. The placement-rule clarification immediately eliminated the ambiguity that had stalled the drift cleanup. Stashing the unrelated `_index.md` change preserved another session's in-progress work without burying it under my commits.

**Didn't work — three categories.**

*Process friction — parallel-CD CWD reset.* When dispatching 5 parallel `Bash` calls each prefixed with `cd <subdir> &&`, subsequent calls landed in arbitrary CWDs. The Bash tool returns to the parent's CWD between calls, but each call's `cd` mutates the parent's *next* default CWD when the prior message had a different `cd` win the race. This caused at least three wasted batches where commands like `head -10 path/to/file` returned "No such file or directory" because CWD was unexpectedly inside a sub-package directory. Fix mid-session: always anchor with `cd /Users/coen/Developer/swift-institute/Experiments && cd <subdir> && cmd`. Not a one-off — happened every time I dispatched parallel Bash with `cd` in each call.

*Diagnostic premature-conclusion — "duplicate" trap.* Saw the same directory name (`api-totality-design`) in both `swift-institute/Experiments/` and `swift-primitives/swift-index-primitives/Experiments/` and initially classified the swift-institute ones as "stale duplicates." Only after the user pushed back ("explain better") did I diff the contents and discover they were **different experiments** with the same name — the .claude cache recovery from Jan 28 had created parallel evolution rather than a duplicate. Almost destroyed work that was the more-actively-developed version. The lesson: when two paths share a name across repos and the natural conclusion is "delete one," diff first.

*Vocabulary blindness.* Spent multiple early-phase batches classifying obviously-FIXED experiments as PASSES because PASSES was the only "everything compiled" verdict in the established vocabulary. Only later, when revalidating `unsafe-forin-release-crash`, did I notice that "documented expected: crash; observed: success" was a third pattern that PASSES doesn't capture. Went back and corrected the v3-with-deps case after the fact. Worth introducing FIXED earlier — it would have surfaced the swiftpm-visionos v3 result as the notable outcome it was, rather than as one more "build clean" line in a commit message.

Two minor process losses: `git add -- $(git diff --name-only)` broke on the `markdown-rendering-performance-profiling/Sources/Capture vs Pure Action Tests.swift` filename (spaces); fix was `git add -u`. And the `add-reval.sh` exit codes (1 for MISSING, 2 for NO_ANCHOR) cascaded into Claude Code's "cancel sibling parallel calls on first failure" behavior several times — chaining helper-script calls with `&&` meant one missing path cancelled four pending experiment runs. Fix: separate helper-script calls from build/test calls in the same parallel batch, never chain them.

## Patterns and Root Causes

**FIXED was missing from the verdict vocabulary because no prior batch had encountered a fixed bug.** The HANDOFF noted "All 42 experiments behave identically to their prior documented results; no regressions surfaced." The previous session had no FIXED case to introduce, so the vocabulary calcified at three verdicts. This is a general pattern: small initial samples bake assumptions about completeness into the vocabulary, and the assumption is invisible until a counterexample arrives. The fix is canonization — once a fourth verdict exists, write it down as a convention rather than leaving it as a one-session improvisation that the next session will have to rediscover.

**The placement rule was implicit ecosystem knowledge that the original handoff alluded to but didn't crisply state.** The handoff said "experiments depending on a package P go in `P/Experiments/`; only standalone (compiler-only) experiments belong here. This session moved 3 in (from swift-io and a loose dir) and 1 out (to swift-parser-primitives) under this rule." That was enough to act on for the cases the prior session encountered, but it wasn't framed as a *rule* — and the ambiguity-handling parts (multiple primitives → swift-primitives umbrella; spans into standards → swift-standards; etc.) only emerged in this session when the user re-stated the rule in response to my drift question. Saving as `feedback_experiment_placement_by_dep_layer.md` makes it discoverable for the next session that hits a placement question.

**The "duplicate" trap is a structural risk created by the .claude cache recovery operation.** The Jan 28 commit "Recover 5 experiment packages from .claude cache" pulled in directories that already existed canonically elsewhere. Three months later, the recovered copies have evolved on their own (swift-institute/api-totality-design became "Array Totality Patterns" while the canonical stayed "Index Totality Discovery"). Anyone iterating through "stale-looking duplicates" without diffing will destroy real work. The general rule: when a name collides across repos, the path that's been *actively edited more recently* is the one most likely to be the divergent fork that needs renaming, not the one to delete.

**Anchor-less experiments were a deferred convention question that didn't actually need deferring.** The handoff treated this as an open action item ("Adding headers retroactively vs leaving blank is a judgment call — the reflection has an action item to standardize this"). When the user simply said "if so, we should add the header," the deferral evaporated in one sentence. The pattern: an open question framed as "needs design discussion" can sometimes be answered by just asking; the cost of asking is much lower than the cost of carrying the question across sessions.

**Parallel-CD CWD-reset is a Claude-Code-Bash-tool gotcha worth a feedback memory.** Every time I dispatched 4–5 parallel Bash calls with `cd <dir> && cmd` in each, subsequent batches landed in arbitrary CWDs. This is a tool-behavior fact, not a project-specific concern. Worth documenting once so future sessions don't rediscover it.
