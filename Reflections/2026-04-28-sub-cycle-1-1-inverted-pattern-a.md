Conservative interpretation of "alongside" — no public typed surface introduced.

**Didn't work**:

- **Wave 1's `@_disfavoredOverload` removal was a misapplication of "match Phase 1.5 Lock.swift / Memory.Map.swift canonical pattern."** The canonical Phase 1.5 shape is "raw with syscall + typed delegating; no `@_disfavoredOverload`." That works for Lock and Memory.Map because *no other layer declares a typed overload at the same name*. IO.Read / IO.Write DO have parallel typed overloads at L3-unifier (swift-kernel cross-platform shims), so the L2 typed forms NEED `@_disfavoredOverload` to defer resolution to L3-unifier. The Wave 1 commit body claimed pattern-matching to Lock/Memory.Map without checking whether the conditions (no L3-unifier shadow) held for IO.Read/Write. They didn't. The fixup grep would have taken seconds: `find swift-kernel/Sources -name "*Kernel.IO.{Read,Write}*"` finds the cross-platform shims; their existence is the trigger for "L2 needs `@_disfavoredOverload`."
- **The pre-flight grep for L3-unifier shadows was implicit, not explicit.** Wave 1 went out without an explicit "are any of these L2 typed names also extended at L3-unifier?" check. The check existed structurally (the swift-kernel build would fail), but only after all 5 waves had landed. A pre-Wave-1 grep would have caught it before any commits. The principal's session-end note codifies this as a future-wave pre-flight requirement; the current session's pattern was reactive, not proactive.
- **Empirical re-scope didn't catch the public-vs-internal asymmetry of File.Delete / Link / Link.Symbolic before planning.** The 16-file count was correct; what wasn't planned-for upfront was that 3 of those 16 files have `@usableFromInline internal` typed forms rather than public ones. The Wave 4 plan accommodated this in-flight (added public `@_spi(Syscall)` raw forms alongside internal typed; typed delegates to raw), but the planning step had defaulted to "all 16 files are public-typed → public-raw alongside" without checking visibility class. Did not surface as a defect because the in-flight accommodation was structurally sound, but the empirical re-scope per [HANDOFF-024] should ideally have flagged the visibility-class asymmetry as a planning input.

**Mixed**:

- **Removing `@_disfavoredOverload` to match canonical pattern.** Reading the Phase 1.5 Lock.swift / Memory.Map.swift pattern as "the canonical shape doesn't carry `@_disfavoredOverload`" was directionally correct (those files don't), but the inference "therefore L2 typed forms shouldn't have `@_disfavoredOverload`" requires the L3-unifier-shadow precondition to be ABSENT. Reading canonical patterns as universal rules (rather than rules with implicit preconditions) is a recurring vector for this class of mistake. The fixup commit's body now documents the precondition explicitly so future readers won't repeat the elision.

## Patterns and Root Causes

**Pattern 1: INVERTED Pattern A as a distinct compositional shape.** Phase 1's earlier sub-cycles (1.2 / 1.3 / 1.4 / 1.5 / 1.6 / 1.7 / 1.8) followed Pattern A: strip the typed L2 form (which had pre-Phase-1 typed `borrowing POSIX.Kernel.Descriptor` parameters), add raw `@_spi(Syscall) fd: Int32` SPI forms in its place, and have the L3-policy compat wrapper in swift-posix carry the typed form. Phase 1.5's Waves 1–7 *re-added* typed forms at L2 alongside the raw forms (originally with `borrowing POSIX.Kernel.Descriptor`, then namespace-corrected to `borrowing Kernel.Descriptor`). Sub-cycle 1.1 is the third compositional shape: the file family was never Pattern-A-stripped — typed forms always existed at L2 in `Sources/ISO 9945 Kernel File/`. Sub-cycle 1.1 ADDS raw `@_spi(Syscall)` forms alongside, refactoring typed to delegate to raw. The shape is "additive raw alongside extant typed; typed delegates to raw via `_rawValue`; raw holds the syscall." The principal's note flagged this as INVERTED Pattern A and reusable for Phase 2 (windows-standard mirror). The reusability is structural: any package whose typed L2 forms exist but whose raw spec-coverage is absent gets the same shape applied — windows-standard's `Kernel.Descriptor` (HANDLE-based) typed forms can have raw `HANDLE @_spi(Syscall)` forms added alongside without altering the typed surface.

The three shapes side-by-side:

| Shape | Trigger | L2 state after | L3-policy compat wrapper |
|---|---|---|---|
| Pattern A (Phase 1 sub-cycles) | L2 had typed; needed raw spec coverage; structural pivot away from typed at L2 | raw only | adds typed wrapper |
| Phase 1.5 retroactive re-typing | L2 had been stripped to raw-only; needed typed back | raw + typed (delegating) | unchanged (still has typed wrapper) |
| INVERTED Pattern A (sub-cycle 1.1) | L2 had typed; needed raw spec coverage; typed retained | raw + typed (delegating) | unchanged or absent (file family is iso-9945-direct) |

Pattern A's L3-policy wrapper layer is the place where EINTR retry / partial-IO loop / error normalization live. INVERTED Pattern A doesn't change that layer — typed and raw coexist at L2; the L3-policy layer (if any) wraps L2's typed form for policy. windows-standard, being single-platform, has no L3-policy layer: cross-package consumers go directly to L2 (typed or raw). This is the "L3 platform tier empty exception" per [PLAT-ARCH-008e].

**Pattern 2: canonical-pattern reading as universal rule vs precondition-bound rule.** The Wave 1 `@_disfavoredOverload` mistake is a specific instance of a general failure mode: reading a canonical shape from prior code as a rule-without-preconditions. The canonical Phase 1.5 Lock.swift / Memory.Map.swift shape is "raw with syscall + typed delegating; no `@_disfavoredOverload`." That shape's correctness depends on a precondition: *no other layer declares a typed overload at the same namespace path*. For Lock / Memory.Map / many other types in iso-9945, that precondition holds. For IO.Read / IO.Write, it doesn't — `swift-kernel/Sources/Kernel File/Kernel.IO.{Read,Write}+CrossPlatform.POSIX.swift` adds parallel typed overloads to the same path. The shape applied without checking the precondition produces ambiguity at every consumer site that imports `Kernel`.

Generalizing: when applying a canonical shape from prior code to a new file, the canonical-shape application MUST include a precondition check. Concrete preconditions for L2 typed-form additions:

- *L3-unifier shadow check*: does `swift-kernel/Sources/.../Kernel.{namespace}+CrossPlatform.{Platform}.swift` declare a typed overload at the same name?
- *L3-policy delegate check*: does `swift-{platform}/Sources/.../Kernel.{namespace}.swift` declare an extension at the same path that may shadow?
- *L2 sibling-platform check*: do other L2 packages (darwin-standard, linux-standard, windows-standard) declare extensions at the same path?

Where any of these is true, the L2 typed form needs disambiguation (`@_disfavoredOverload` to defer; or rename to avoid). Where none is true, the canonical shape applies directly. The pre-flight is mechanical: a `find swift-foundations/swift-kernel/Sources -name "Kernel.{namespace}*"` or `grep -rn "extension Kernel.{namespace}"` finds the parallel declarations.

This pattern recurs whenever a layer adds extensions to a shared namespace. The 2026-04-20 reflection on `Kernel.Error.Code` predicates hit a related shape — extensions at a shared namespace path can collide with extensions at nominally-different paths that resolve to the same place via typealias chain. The L3-unifier-shadow case is the simpler instance: same literal path, same module-resolution at compile time, ambiguity manifests as the language's overload-resolution diagnostic.

**Pattern 3: pre-flight prereq is conditional on import-graph density, not universal.** The principal's session-end note flagged this directly: sub-cycle 1.7 needed a Linux baseline prereq (`f2959bc / 6bb1402`) before its main work because the file family had cross-package vestigial imports (shm_open + Thread.Key). Sub-cycle 1.1 ran the same pre-flight and needed nothing — the File family's typed forms are self-contained within iso-9945's `ISO 9945 Kernel File` target. The prereq is conditional on the file family's import-graph density: families with cross-package imports surface latent breaks; families with target-internal imports don't.

This means the [HANDOFF-024]-style pre-flight rule is "MANDATORY to run; CONDITIONAL on producing a prereq commit." The current handoff text's framing of "Linux baseline audit pass before first wave (analog to 1.7 prereq)" is correct as a process step but ambiguous as a deliverable — a green pre-flight is a successful pre-flight; only a red pre-flight needs a fix-up commit. The session's experience confirmed: pre-flight ran clean → no prereq commit → Wave 1 dispatched directly. The conditional outcome is the right model; codifying it as such in the handoff skill (or the platform skill's pre-flight guidance) prevents future sessions from hesitating on "do I need a prereq commit if pre-flight is clean?"

**Pattern 4: per-wave isolation contains scope of fixups; downstream verify is the final integration gate.** The 5-wave decomposition meant each wave's commits were green on iso-9945 in isolation. Wave 1's `@_disfavoredOverload` defect didn't surface in iso-9945's own builds because iso-9945 doesn't have downstream-build awareness — it can't see that swift-kernel's test target consumes `Kernel.IO.Read.read(...)` ambiguously. The downstream verify step (running swift-kernel, swift-memory build at end-of-cycle) is the integration gate. Per-wave isolation contained the fixup scope to Wave 1's two files; the alternative (single big commit) would have required the same fix but with a larger blast radius for any other defects discovered.

**Root cause synthesis (Patterns 1–4)**: *the L3-unifier and L2 are at the same namespace path; when L2 adds typed forms, it must consider L3-unifier as a parallel author at the same path*. This is a structural reality of the platform stack's typealias-chain composition: L3-unifier `swift-kernel` re-exports L2 namespaces and adds its own typed overloads at the same path, expecting consumers to see "the unified surface." When L2 adds new typed forms (or removes `@_disfavoredOverload`), it changes the unified surface in a way the L3-unifier didn't anticipate. The pre-flight grep for parallel L3-unifier extensions is the mechanical check that closes this. INVERTED Pattern A makes this more salient because it's purely additive at L2 — but the additive-ness is exactly what creates new overload-resolution interactions with extant L3-unifier authors.

The principal's note about pre-flight grep for `@_disfavoredOverload` on the L2 side is the codified version of this. Phase 2 (windows-standard) will hit the same pre-flight check, but the L3-unifier shims are POSIX-side only (`Kernel.IO.{Read,Write}+CrossPlatform.POSIX.swift`); windows-standard adds at `Windows.Kernel.IO.X` (which typealiases to `Kernel.IO.X` per [PLAT-ARCH-004]) and may or may not collide depending on whether the L3-unifier has Windows-side counterparts. The check is the same: grep `swift-kernel/Sources/` for cross-platform shims at the windows-standard typed-form names; if present, `@_disfavoredOverload` needed.
