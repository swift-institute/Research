## What Worked and What Didn't

**Worked well:**
- Parallel subagent dispatch at scale. 5 subagents in parallel for tier-3 was the largest dispatch in any taxonomy tier; total wall-clock ~6.5 min. Cluster sizing by ID count + line count produced reasonable load balance.
- The stash-edit-commit-pop pattern for cross-session contamination. Used in both tier-2 and tier-3; both pops cleanly auto-merged. Cleanest mechanism I've found for "stage only my hunks" without `git add -p` interactivity.
- Pre-flight escalation in tier-3. The brief explicitly added a pre-flight enumerate-symlinks check; surfacing the 9 engagement+ingest skills as a finding (NOT classifying them) was the correct read of the MUST/ask interaction. Supervisor block discipline held.
- Cross-skill ID overlap detection in tier-2 (conversions ↔ index 15 shared `[IDX-*]` rules with full bodies in both files). This was an emergent finding — not predictable from the brief; surfaced by reading both classification tables and noticing duplicate IDs. Documented as implicit Composite signal in §10.
- Final-inventory section in tier-3 §28. The brief explicitly required ecosystem-wide totals + mechanical-rule construction queue. Producing both as a single ranked table that calls out γ-roadmap match status creates a directly-actionable workflow-construction substrate.

**Didn't work well:**
- **Subagent γ-mark force-fitting was a recurring quality failure.** Both tier-2 (cluster A invented γ-N3) and tier-3 (clusters B and E applied γ-1a/γ-2b/γ-3/γ-3b to rules outside v1.2.0's actual γ-class scope) produced creative but wrong γ classifications that I had to strip during synthesis. The brief's "do not force-fit" directive was insufficient; subagents extended γ classes by analogy ("file-shape rules are like target-fidelity laboratories because both are deterministic," etc.). The cost: ~95 force-fit marks across the two dispatches required regex-strip + density-prose-rewrite passes during synthesis.
- **Subagent self-correction noise.** Both dispatches had clusters where the agent's distribution table contained self-corrections ("I listed 12 IDs but I should have listed 13, corrected below"), inflating doc length. Tier-3 cluster B's skill-lifecycle classification correctly identified that 2 of the `### [` heading hits were template-literal placeholders inside an example block (not real rules), but only after a self-correcting note — the count `30 real rules vs 32 grep-hits` is now a documented finding. Cluster A's readme classification had a similar reconciled-distribution note.
- **`memory-safety` and `readme` and `implementation` hub-with-sibling-files structure produced ambiguous scope.** Tier-2's memory-safety has 57 IDs across hub + 8 siblings; tier-3's readme has 83 across hub + 6 siblings; tier-3's implementation has 4 hub axioms + ~70 corollary rules in siblings (out of scope per brief). The "navigate hub vs full corpus" decision was made differently for each: memory-safety walked all 57, readme walked all 83, implementation walked only the 4 hub axioms. The brief's "only count IDs declared with `### [` headings in this file" clause for implementation was load-bearing — without it, implementation's classification would have been ~70 rules instead of 4. This inconsistency is real and surfaces a corpus-classification design ambiguity.
- **Brief calibration vs actuals**: tier-3 brief expected "dominantly semantic" voice/process distribution; actual was 48% mechanical, comparable to tier-2's 51%. The semantic floor (supervise 26%, research-process 16%) is offset by operational sub-clusters (testing-* 80–100%, dutch-law/legal-encoding 55–86%). The brief's expectation was wrong; surfacing this as calibration data in §26.3 was the correct response, but the wrong expectation propagated through the brief's framing ("voice/process skills") which is a less precise descriptor than the actual content.

## Patterns and Root Causes

**Subagent γ-mark force-fitting** (and the analogous "non-existent class invention") is a structural property of taxonomy delegation, not a one-off failure. The pattern: when a subagent has a rule body in front of it that fits the *operational shape* of an existing class (deterministic scan, schema audit, file-presence check) but isn't in the *named scope* of that class (v1.2.0 §3.4.5's γ-2 targets `.github/workflows/**`, not arbitrary file-shape audits), the subagent reaches for the most analogous existing class rather than reporting "no match." The prompt's explicit "do not force-fit" doesn't override the analogical-completion bias because the bias activates at the level of "this rule looks like deterministic scan" before the prompt's negation applies. The fix is structural: the prompt must enumerate **valid γ-class names verbatim** AND state that any rule outside the enumerated set should be reported with no γ mark. The negative formulation ("do not force-fit") is too weak; the positive formulation ("only γ-1a, γ-1b, γ-1c, γ-2, γ-2b, γ-3, γ-3b, γ-4 are valid; report unmarked otherwise") closes the gap. This generalizes: any taxonomy delegation prompt SHOULD enumerate valid classes and forbid invention, not merely caution against forcing.

**Cross-skill ID overlap as a corpus-coherence signal** is a finding I would not have caught without classifying both skills back-to-back in the same dispatch. The conversions ↔ index 15-rule overlap is a real corpus-coherence issue (same rule declared with full bodies in two skills, with 2 of 15 classifying differently from each skill's vantage). The pattern: when a skill is named for a domain (`index` for typed-index types) and a sibling skill is named for an operation across that domain (`conversions` for typed-value conversions), the operation-skill ends up duplicating the domain-skill's definitional anchors as predicates for its own rules. This is an emergent finding; the briefs didn't anticipate it. Generalization: tier-3 likely has analogous overlaps (e.g., `testing` and `testing-institute` and `testing-swiftlang` may share ID-prefix surface) — a cross-skill overlap audit could be a separate dispatch using the now-classified corpus as input.

**The "navigate hub vs full corpus" design ambiguity** for hub-with-siblings skills (memory-safety, readme, implementation) is unresolved. The current handling (each brief decides per-skill what counts) produces an inconsistent corpus-coverage standard: memory-safety's 57 sibling-file rules are classified, readme's 83 are classified, but implementation's ~70 are not. Future dispatches need an explicit rubric: do navigation-hub skills always classify hub + siblings, or only the hub? If siblings, what's the file:line page-cite shape? The implementation skill's case is special — its hub axioms are *governing principles* whose corollaries in sibling files are *concrete rules*; the hub-axioms-only classification is defensible because the corollaries are scoped by the axioms, not independent.

**Brief calibration drift** (tier-3 "dominantly semantic" expectation vs 48% actual) is normal — briefs encode the parent's prior beliefs about each tier's character; actual classifications calibrate them. The pattern: voice/process is a description of the skill's intent, not its requirement-rule shape. A skill can be "voice/process" by intent (capturing a workflow) while having a mechanical-heavy rule body (the workflow's artifacts have schemas). testing-institute is the canonical example: its intent is "set up nested test packages for dependency isolation" but its rule body is 100% Package.swift / filesystem / AST shape. The brief's framing axis (intent) and the classification's framing axis (rule-body shape) are orthogonal; conflating them produces calibration drift.

**The stash-edit-commit-pop pattern** for cross-session contamination handling worked cleanly twice in this session. It generalizes: any session that needs to commit a partial diff in the presence of pre-existing uncommitted working-tree changes from prior sessions can use this pattern. It's currently not codified in either the handoff skill or the reflect-session skill. Codification value: the pattern is non-obvious (requires `git stash push -- <files>` syntax for selective stash, then `git stash pop`'s auto-merge handling) and would otherwise require re-derivation in each session that hits the situation.

**Final-inventory as a "closing arc" pattern.** Tier-3's §28 closes a goal-thread that began with the pilot. The shape — ecosystem-wide totals + ranked construction queue + γ-roadmap match status + latent-reservoir identification — is reusable for other multi-dispatch sweeps. The construction queue's ranking is the actionable artifact; without it, the 1045-rule corpus is too large to prioritize. The pattern: when a tiered sweep closes, the closing dispatch SHOULD produce a final-inventory section that aggregates across all tiers and ranks the actionable units (here: skills by mechanical rule count) for the downstream phase (here: workflow construction).
