## Patterns and Root Causes

**Pattern 1: "Lowest-risk diff" vs "structurally correct" as recommendation axes in Doc-class investigations**. When an investigation doc presents an options matrix, the default evaluation axis should be structural correctness (which option recognizes the actual type/shape structure?), not diff-size or risk-cost. Diff-size is a tiebreaker for structurally-equivalent options, not a selection criterion over structural options. In P2.9, Option A and Option B were structurally distinct: A recognized A-and-B as the same type-with-two-initializers; B reified their accidental historical split as two types. The right axis was structural. My draft Doc 4 used diff-size as a recommendation axis when it was only appropriate as a tiebreaker — a framing error.

This pattern generalizes to Doc 1/2/3 and future Doc-class investigations. When writing an options matrix, the recommendation section should state *the structural goal* first, evaluate each option against it, and only then consider secondary axes (diff-size, risk-cost, migration-scope). When structural and secondary axes conflict, structural wins unless the principal explicitly invokes a secondary axis for a specific reason. Cycle 3's Doc 1 had the same pattern: Shape-vs-Decisions coherence required Decisions to override Shape when they conflicted (reflection `2026-04-24-cycle-3-close-and-inlinable-spi-cascade-non-fire.md`). Both cases are instances of the same meta-rule: when framing diverges from substance, substance wins.

**Pattern 2: Grep-based audits miss module-import visibility under Swift 6.3+ strictness features**. `InternalImportsByDefault` + `MemberImportVisibility` introduced a new class of audit gap: a type's *visibility* at a consumer depends on whether the consumer has explicitly `public import`ed the defining module, not on whether the type is referenced by name anywhere. Grep finds text matches; the compiler enforces import relationships. Pre-Swift-6 ecosystems had lax transitive visibility through umbrella re-exports that made grep-level consumer counts approximately correct. Swift 6.3+ ecosystems using the new strictness features narrow transitive visibility, so accidental visibility (e.g., B's co-location in `Linux_Kernel_System_Standard` covering the io_uring consumer's needs) can mask the true import relationships until deletion exposes them.

The fix class: any Doc-class audit that proposes deleting-without-adoption of a publicly-declared type SHOULD add a build-level pre-flight verification. Candidate mechanisms: (a) `swift symbolgraph-extract` on each consumer module to enumerate visible types, (b) trial-deletion-plus-build at the principal consumer sites to surface hidden-import gaps before the real deletion commits, (c) a mechanical grep for `public import Module_Of_Type_Being_Deleted` at each named consumer site. Phase 0.5's methodology in Doc 4 had (a) implicitly assumed from the zero-consumer grep result but didn't verify (c). Option (c) is the cheapest Phase-0.5b addition; Option (a) is the most rigorous. Worth documenting in `handoff` skill's `[HANDOFF-013a]` Writer-Side Prior-Research Grep as a sub-rule.

**Pattern 3: Toolchain branding-version is not toolchain behavior**. `swift --version` reporting `6.3.1` across Apple's Xcode build and swift.org's 6.3.1-RELEASE are different compiler builds — different codegen, different strictness enforcement, different generic-resolution paths in some cases. Target-architecture differences within swift.org's own 6.3.1-RELEASE (macOS ARM64 vs Linux ARM64) yield further behavior differences; D4 is one empirical instance. Cross-platform verification that relies on "same Swift version string" equates version equality with behavior equality — which it doesn't, especially under `StrictMemorySafety` + `InternalImportsByDefault` + `MemberImportVisibility` + nested-generic-type-alias resolution paths.

For the ecosystem: tooling choices should standardize on actual build IDs (swift.org's tarball / canonical release sources), not version strings. `swiftly` is the correct abstraction on macOS; Docker's swift:6.3.1 image is the canonical Linux reference. The earlier session-default of using `/usr/bin/swift` (Apple Xcode) for macOS gates was a false-positive source that Cycle 4's D4 re-diagnosis finally exposed.

**Pattern 4: Docker-on-macOS wedge modes outside Docker's process tree**. Docker Desktop on Apple Silicon hosts the Linux guest in an Apple VZ framework VM. The daemon socket is accepted by Docker's `com.docker.backend`, but the VM lifecycle is managed by Apple's `com.apple.Virtualization.VirtualMachine.xpc`. Long-running sessions doing repeated swift:6.3.1 container builds that bind-mount `/Users` can exhaust the VM's fd table via virtiofs passthrough (observed: 59,212 fds after 8d14h uptime). "Restart Docker" doesn't fix it because the leaker is Apple-owned; only menu-bar Quit or a targeted `pkill` on the XPC process recovers.

Recurrence: any session doing repeated Docker Linux container builds with `/Users` bind-mount is at risk. Mitigation candidates: (a) restart Docker Desktop between multi-repo Linux gate batches (preventive), (b) switch to VOLUME-mount instead of bind-mount (changes the mount semantics but avoids virtiofs passthrough accumulation), (c) document the wedge signature + recovery so future sessions resolve it in 30 seconds via menu-bar Quit instead of 4 minutes of retries. The HANDOFF-docker-desktop-unresponsive.md on disk is the durable reference; a future recurrence should read it first. Worth considering as a tooling-hygiene `[skill]` update — possibly a memory note on recovery steps.

**Pattern 5: Cross-cycle latent ambiguity surfaces at broader-scope gates; broader-scope gates are likely to be blocked on unrelated drift**. P2.9 was latent through Cycles 1–3 because their gates were tighter than the configuration exposing the redeclaration. Cycle 3's gate (d) was the first broad-enough Linux Docker gate; it was blocked by the same-name redeclaration it was meant to surface. The post-Cycle-3 re-verification explicitly found P2.9; Cycle 4's terminal gate was similarly blocked on D5 (unrelated upstream Swift 6.3 `MemberImportVisibility` drift). The substantive-canary pattern (Phase 2's narrower Docker Linux `--build-tests` carrying the gate's intended load) closed both cycles.

The meta-pattern: multi-cycle arcs accumulate cross-cycle interactions that only surface at the broadest gate; the broadest gates are the most likely to be blocked on unrelated ecosystem drift. The mitigation is not "fix the broad gate at all costs" (fighting losing-battle ecosystem drift) — it's "design narrower substantive canaries whose coverage overlaps the broad gate's intended scope, and codify canary substitution as a first-class termination path." Cycle 3 gate (b)/(d) established the precedent; Cycle 4 inherited and extended it. This is worth a `[research]` topic on canary-gate substitution methodology for future long-arc work.
