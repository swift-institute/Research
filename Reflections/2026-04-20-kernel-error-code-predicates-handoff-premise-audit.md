## Patterns and Root Causes

**The premise-audit gap.** The handoff asserted a factual precondition — "swift-windows-standard compiles on Windows only" — as the justification for "zero `#if os(...)`." My pre-flight verified everything around the instruction: the cited rule `[PLAT-ARCH-008c]` was loaded, collisions were checked, the precedent was read, constants were counted, audit staleness was verified per `[HANDOFF-016]`. I read the Windows Package.swift, saw no platform condition on Windows Kernel Standard Core, noted in internal monologue that "the target IS being compiled on macOS," and then *deferred* the implication. The pattern: **a handoff's factual preconditions are not the same as its instructions.** When an instruction is "do X because of Y," and Y is a checkable claim, Y is the thing to verify first — not X. I verified X (the instruction ran through on the swift-kernel path) and hoped Y (the Windows-only-compilation premise) was true.

`[HANDOFF-016]` covers Next-Step staleness and proposal staleness; the sister notion for **premise staleness** — the handoff's factual basis no longer matching reality — is absent from the skill. This is the same family as the scope-flag staleness I named in the earlier `2026-04-20-file-name-nul-fix-execution-and-path-char-adoption.md` reflection: both are cases where the handoff's *framing* is stale, not its *work items*. Scope-flag staleness (session discovers the scoped-out item is in fact the root cause) and premise staleness (session-verifiable claim is false) are two points on the same axis — the axis the current skill does not cover.

**The permission-read vs. preference-read pattern.** "If no precedent tests exist, that's acceptable" has two readings: (a) permission to skip whenever the precondition fires, (b) noting where the floor is for unusual cases. I read (a); the author meant (b). This is the same family as the `[PLAT-ARCH-008a]` confusion in the earlier reflection: I answered "is this permitted?" when the question was "is this necessary?" The sentence shape differs; the error is identical. My default is the permission-read; the correct default is the preference-read, because opt-outs are softer-than-rules and exist to cover unusual cases — not to define the expected path.

A concrete mechanical check: when an instruction carries an opt-out with shape "if P, then X is acceptable," ask "is my situation the class of case the author had in mind when writing the opt-out?" rather than "does P hold?" The former reads intent; the latter reads the literal trigger. For a 2-line-wrapper predicate test, the answer to the former is "no" even though the answer to the latter is "yes."

**The rule-vs-tooling gap.** `[PLAT-ARCH-008c]` says "the package boundary IS the conditional." The principle is sound in the platform-stack layering model. SwiftPM 6.3 has no mechanism to realize it at the target level — `.target(..., condition: .when(platforms:))` does not exist; conditions apply only to dependencies and product consumers. The actual realization is "file-body `#if os(X)` guards in every file of the platform-specific target," which is what `Windows.Kernel.Error.swift`, `Windows.Kernel.swift`'s descriptor veneer, and now `Windows.Kernel.Error.Code+Predicates.swift` all use. The rule describes an ideal that Swift 6.3's package format cannot produce; when implementers follow the rule literally, they produce broken configurations (my first Windows file). The rule should either explicitly codify the file-body guard convention as the intended mechanism or acknowledge the gap and reference an external convention doc.

**The reviewer-as-frame-shifter pattern.** The tests correction wasn't a test-coverage concern — it was a re-reading of the handoff's opt-out clause from a frame I hadn't occupied. The handoff author and the reviewer were the same mind at different times; my session read the clause literally, the reviewer read it against the author's remembered intent. The Windows-build failure likewise surfaced from a frame-shift: the test harness built `Windows Kernel Standard Core` directly on macOS, a frame the swift-kernel-path build never occupied. In both cases, the correction came from re-reading the situation through a lens my execution did not naturally adopt. The asymmetric skill is *reframe-and-re-read* vs *execute-and-verify*; the latter is necessary but insufficient on its own. `file-name-nul-fix-execution-and-path-char-adoption.md` named this pattern for the supervisor/user pair; this session extends it to the handoff-author pair and the build-harness pair. The unifying observation: a second frame is almost always available (different reader, different build path, different platform, different question), and first-pass implementation systematically fails to adopt the second frame on its own.

## Action Items

- [ ] **[skill]** handoff: Add a "precondition audit" rule as sister to `[HANDOFF-016]` proposal staleness. When a handoff cites a factual claim as justification for an instruction (e.g., "swift-windows-standard compiles on Windows only, therefore zero `#if os(...)`"), the implementer MUST verify the claim against current configuration (Package.swift, code state, build output) before executing the instruction that depends on it. Premise staleness is a distinct category from Next-Step staleness or proposal staleness — it covers the handoff's factual basis, not its work items. Pairs with the scope-flag-staleness pattern named in `file-name-nul-fix-execution-and-path-char-adoption.md`.
- [ ] **[skill]** platform: Clarify `[PLAT-ARCH-008c]`'s "the package boundary is the conditional" mechanism. SwiftPM 6.3 has no target-level platform condition — `.target(..., condition: .when(platforms:))` does not exist; conditions apply only to target dependencies and product consumers. The actual realization for platform-specific L2 targets (Windows Kernel Standard Core, ISO 9945 Core, Linux Kernel Standard Core) is file-body `#if os(X)` guards, as `Windows.Kernel.Error.swift` and `Windows.Kernel.Error.Code+Predicates.swift` both use. The rule should (a) codify the file-body guard as the intended mechanism, and (b) note the asymmetry: the POSIX branch needs no guard because POSIX constants are available on all platforms the ISO 9945 Core target builds on, while the Windows branch does need a guard because Windows constants are not available on Apple platforms.
- [ ] **[skill]** handoff (or a new handoff-reading skill): Document the "opt-out as preference vs. permission" distinction. When a handoff (or skill rule) offers a conditional opt-out of shape "if P, then X is acceptable," the implementer SHOULD read the opt-out as expressing the author's preference for the unusual-case class, not as blanket permission whenever P holds. Concrete mechanical check: ask "is my situation the class of case the author had in mind?" not "does P fire?" Applies whenever a handoff or rule has softer clauses like "acceptable to skip," "can defer if," "optional when." This generalizes the `[PLAT-ARCH-008a]` "is this permitted?" vs. "is this necessary?" distinction named in the earlier reflection.
