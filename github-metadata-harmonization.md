## Summary

**Current state.** Across 17 GitHub orgs (the 11 enumerated by `Scripts/sync-community-health.sh` plus `swift-linux-foundation`, `swift-microsoft`, `swift-arm-ltd`, `swift-intel`, `swift-riscv`, and the `swift-institute` host org itself) there are **439 repositories** (102 public, 337 private, 14 archived). On the public surface — the only surface where metadata is load-bearing — coverage is highly uneven: descriptions on **88/102** (86 %), topic tags on **13/102** (13 %), homepage URL on **47/102** (46 %, and 47 of those 47 point at the wrong host — the maintainer's personal `https://coenttb.com`), and license auto-detected on **80/102** (78 %). Description style varies between four distinct shapes within the same org; topic vocabulary is entirely free-tagged with no shared taxonomy; the dominant homepage value is a personal URL that does not represent the Institute. Side-fields are uniform (`hasIssuesEnabled=true`, `defaultBranchRef.name=main`) except for one experimental-branch outlier and inconsistent wiki/discussions toggles.

**Proposed standard.** A single per-package `.github/metadata.yaml` file in each repo's `.github/` directory carries the canonical description / topics / homepage / settings and is read by reusable workflows centralized in `swift-institute/.github` that propagate it to GitHub via `gh repo edit`. Description format is layer-typed: L1 primitives use a noun-phrase Apple-style description; L2 standards use the existing `Swift implementation of {SPEC-ID}: {Title}.` shape (already the dominant pattern in `swift-ietf` and `swift-iso`); L3 foundations use a concise composition phrase. Topic tags follow a fixed taxonomy under Apple's redundancy-omission convention: bare layer tag (`primitives` / `standards` / `foundations` / `components`, no `swift-` prefix) + authority + spec-id where applicable + 1-3 domain tags from a curated registry; the `swift` tag and the `swift-` layer prefix are forbidden as redundant. Homepage URL is the per-package SPI documentation URL when the package is on Swift Package Index; `https://swift-institute.org` otherwise. License auto-detection is enforced via a top-level `LICENSE.md` file on every PACKAGE repo (org `.github` community-health repos are exempt). `hasIssuesEnabled=true`, `hasDiscussionsEnabled=false`, `hasWikiEnabled=false`, `defaultBranchRef.name=main`.

**Tooling.** Three reusable workflows in `swift-institute/.github/.github/workflows/`: `sync-metadata.yml` (the workhorse, takes `org`/`repo`/`dry-run` inputs), `sync-metadata-nightly.yml` (cron 04:00 UTC + manual dispatch — the autopilot), `metadata-pr-preview.yml` (auto-comments PRs touching `.github/metadata.yaml` with the proposed sync diff). Per-repo callers — generated by the existing `sync-ci-callers.sh` fan-out — fire on PR (preview) and on push-to-main (immediate per-repo sync), so convergence happens within seconds of merge. A `generate-metadata.yml` workflow drafts `.github/metadata.yaml` PRs from heuristics (Package.swift `description:` + spec-id pattern-matching) for the ~70 % of the public surface where authoring is mechanical.

**Rollout plan.** Phase 0 publishes this standard and the new `/github-repository` skill (the recommendation in § 6b below) for principal review. Phase 1 lands the three reusable workflows + the per-repo caller template in `swift-institute/.github` and re-runs `sync-ci-callers.sh` to fan the caller out to all public repos. Phase 2 dispatches `generate-metadata.yml` against `swift-ietf` (45 public repos) — the generator opens N PRs with heuristic-seeded YAMLs; principal reviews and merges; on-merge sync converges each repo automatically. Phase 3 fans out to the remaining orgs the same way. The 47 `coenttb.com` homepage values and the one `experiment/typed-algebra-dsl` default-branch outlier are remediated in the same pass. Once steady-state, every public flip just adds `.github/metadata.yaml` as part of the flip-prep PR; the autopilot handles the rest.

## § 3. Proposed standard

A field-by-field standard with rationale, exemplars from § 1, and counter-examples drawn from the inventory.

### 3.1 Description format

**Statement.** Every public repo MUST carry a description that fills exactly one of the templates below. Template selection is mechanical from the package class (which is itself mechanical from the org + repo name). The maintainer's only authored input is the **slot value** — the rest of the description is fixed text rendered by the template.

This makes description authoring a fill-in-the-blank exercise rather than a writing exercise. A workflow can render the description deterministically; a validator can reject any description that doesn't match its template.

#### 3.1.1 Templates by package class

Each template is given as `{TEMPLATE TEXT}` with `«SLOTS»` marked. The slot is the only authored input. Everything outside the slots is fixed.

| Package class | Detected by | Template | Slot definition |
|---|---|---|---|
| **L1 primitive** | repo name `swift-{noun}-primitives` in `swift-primitives/` | `«Content phrase» for Swift.` | `«Content phrase»` = a noun phrase naming the types, protocols, or values the package exposes. **Do NOT repeat "primitives"** — it's already in the package name and the `primitives` topic tag (Apple convention; `swift-collections` says "data structures", not "collection primitives"). Domain-named packages keep it short ("Buffer types and allocators", "Time and date types"); capability-protocol packages name the protocol AND its purpose ("The Carrier super-protocol for phantom-typed value wrappers"). Length 35-100 chars typical. |
| **L2 single-spec** | repo name `swift-{authority}-{n}` in a body org (`swift-ietf` / `swift-iso` / etc.) | `Swift implementation of «AUTHORITY-FULL» «N»: «Spec Title».` | `«AUTHORITY-FULL»` = mechanical from authority code via the table below. `«N»` = mechanical from the repo name. `«Spec Title»` = looked up in `swift-institute/.github/spec-titles.yaml` (§ 3.1.3); if absent, maintainer authors directly and PRs the title into the lookup table in the same wave. |
| **L2 meta-package** | repo name `swift-{family}-standard` combining ≥ 2 specs | `«Family» meta-package combining «Specs» in Swift.` | `«Family»` = the standard's family name (e.g., `IPv6`, `IPv4`, `Email`). `«Specs»` = comma-separated authority + numbers, Oxford-comma'd if 3+ (e.g., `RFC 4291, 4007, and 5952`). |
| **L2 named standard** | repo name `swift-{authority}-{name}` where `{name}` is non-numeric (e.g., `swift-whatwg-html`, `swift-w3c-cssom`, `swift-rfc-3986`-style with named subspec) | `Swift implementation of «AUTHORITY-FULL» «Standard Name».` | `«Standard Name»` = the standard's canonical name verbatim from its publication (e.g., `HTML Living Standard`, `URL Standard`, `CSSOM`). |
| **L3 foundation** | repo name `swift-{noun}` (or `swift-{a}-{b}`) in `swift-foundations/` | `«Capability phrase» for Swift.` | `«Capability phrase»` = 2-6 word noun/verb phrase in sentence case, names what the package DOES (capability), not what it IS (type). E.g., `HTML rendering and tree types`, `PDF rendering`, `Socket networking`, `Time and date handling`. |
| **L3 integration** | repo name `swift-{base}-{addon}` where the package integrates `{addon}` with `swift-{base}` | `«Addon» for swift-«base».` OR `«Capability» integration for swift-«base».` | `«Addon»` = the integrated library's brand name (e.g., `FontAwesome icons`, `Chart.js charts`, `Prism syntax highlighting`). `swift-«base»` = the base package being integrated with, kebab-case. |
| **L4 component** | repo name in `swift-foundations/` or `swift-components/` (when that org exists) | `«Capability phrase» for Swift.` | Same as L3 foundation; component scope is broader but the template is identical. |
| **L5 application** | repo name in a commercial org | Free-form sentence ending in period. | Out of scope for the Institute's deterministic template set. |
| **Org `.github`** | repo name `.github` in any org | `Organization-level community-health defaults for «OrgName».` | `«OrgName»` = the org's display name, derived mechanically (`swift-primitives` → `swift-primitives`; `swift-institute` → `Swift Institute`). |
| **Org website-stub** | repo name `swift-{layer}.org` in `swift-{layer}/` | `Stub for the future «Layer».org website. Content will be developed; for now, see https://swift-institute.org.` | `«Layer»` = the layer name verbatim from the repo (`swift-primitives.org` → `swift-primitives`, etc.). |

For the existing 50+ `swift-rfc-N` / `swift-iso-N` / etc. public repos, Phase 1 of the rollout pre-populates the table by harvesting titles from the existing IETF / ISO descriptions in the inventory (most of which already follow the template).

#### 3.1.4 Common rules across all templates

- Terminal period. No exceptions.
- Third-person, no first-person pronouns.
- MUST NOT start with "A Swift package for…" (per [README-006]'s opening-contract clause).
- MUST NOT include marketing language without substance ("the best…", "blazing fast…").
- MAY include quantified claims ("processes 1,939 PDFs/sec", "zero-allocation in steady state") where relevant and current — only as a *suffix* to the rendered template, never replacing it.
- SHOULD mirror the package's `README.md` first-paragraph one-liner per [README-006] when both exist; the description IS the one-liner truncated to fit GitHub's 350-char limit (the templates here ARE one-liners; this rule operates in reverse: the README opens with the same template-rendered phrase).

#### 3.1.8 Rendering pipeline

The `generate-metadata.yml` workflow (§ 4.4) renders descriptions by:

1. Detecting the package class from `{org}/{repo}`.
2. Selecting the matching template.
3. Filling slots from the rules above (lookup tables for `«AUTHORITY-FULL»`, `«Spec Title»`; mechanical extraction for `«N»`, `«Layer»`, `«OrgName»`; maintainer-authored values from a small `.github/metadata.slots.yaml` companion file OR inline overrides in the YAML for the few cases where the slot can't be mechanically derived).
4. Emitting the rendered description into the `.github/metadata.yaml` PR.

For ~70 % of the public surface (spec-id-shaped repos + org meta-repos + website stubs + institute meta-repos), description rendering is fully automatic. For the remaining ~30 % (L1 primitives with non-trivial taglines, L3 foundations with hand-authored capability phrases), the maintainer fills one slot value and the workflow renders the rest.

### 3.2 Topic tags

**Statement.** Every public repo MUST carry topic tags. The taxonomy follows Apple's redundancy-omission convention (resolved per § 7 Q2): tokens already implicit in the repo name or the org are NOT repeated as tags.

| Category | Required | Values | Source |
|---|---|---|---|
| Layer | MUST (1) | `primitives` / `standards` / `foundations` / `components` | from package class |
| Authority | MUST when applicable (1) | `rfc` (any IETF), `iso`, `ieee`, `iec`, `w3c`, `whatwg`, `ecma`, `incits` | from package's spec authority |
| Spec ID | MUST when applicable (1) | `rfc-{n}`, `iso-{n}`, `ieee-{n}`, `incits-{n}-{year}`, `ansi-x3-4`, etc. | from package name |
| Domain | MUST (1-3) | from § 3.2.2 curated registry | from package's primary purpose |
| Design property | MAY (0-3) | `type-safe`, `noncopyable`, `pure-swift`, `swift-embedded`, `cross-platform`, `concurrency-safe`, `sendable`, `zero-allocation` | from package's notable design properties |
| Brand | MAY (0-1) | `swift-institute` | when packages should be co-discoverable as Institute artefacts (rare; the layer + org context already does this) |

**No `swift` tag.** Apple's `swift-collections`, `swift-system`, `swift-numerics`, `swift-atomics` all omit `swift` from topics — the package name already says it, and the org context reinforces it. Following the same convention, ecosystem packages MUST NOT include `swift` as a topic. (Apple's own practice is mixed; we converge on the redundancy-omission case as the canonical form.)

**No `swift-{layer}` tag.** The layer tag is the bare form (`primitives`, not `swift-primitives`) — the `swift-` prefix repeats the org name. The previously-observed `swift-primitives` / `swift-standards` topic-tag uses (3 occurrences in the inventory) are wrong and migrate during the rollout.

**Range**: 3-10 topics per repo. GitHub allows ≤ 20 per repo and ≤ 50 chars per topic.

**Format rules** (from GitHub's topic constraints):

- Lowercase, alphanumeric + hyphen, ≤ 50 chars.
- Spec IDs use kebab-case with the spec authority's natural identifier (e.g., `rfc-5234`, not `rfc5234`; `ieee-754`, not `ieee_754`; `incits-4-1986` matching the existing `swift-incits-4-1986/swift-incits-4-1986` pattern).
- No `swift-package` or `spm` (redundant); no `swift5` / `swift6` / `swift7` (these go stale); no `swift` (redundant per Apple convention above).

## § 4. Tooling proposal — autopilot, centralized in `swift-institute/.github`

The standard's automation lives entirely in `swift-institute/.github` (the canonical centralized-CI repository) as a small set of reusable workflows. No standalone shell scripts in `Scripts/`; the workflow YAML *is* the implementation. Per-repo workflow files are thin callers generated by the existing `sync-ci-callers.sh` infrastructure (no per-repo hand-authoring). The maintainer's only routine input is reviewing and merging `.github/metadata.yaml` PRs.

### 4.1 Canonical metadata source: per-package `.github/metadata.yaml`

#### Purpose

Move the metadata source-of-truth from GitHub (where edits happen via the web UI or `gh repo edit` and are invisible to git history) into a file in the repo (where edits happen via PR, are reviewed, and live in git history alongside the code they describe). The principal stops being responsible for "remembering to update GitHub when the package changes"; the YAML edit is the trigger and convergence is automatic.

#### Path

`.github/metadata.yaml` (resolved 2026-04-29). The `.github/` directory is GitHub's idiomatic location for repository-level configuration: `.github/dependabot.yml`, `.github/FUNDING.yml`, `.github/pull_request_template.md`, `.github/ISSUE_TEMPLATE/*` already live there. Metadata fits the same pattern. Root-level placement (`.github-metadata.yaml`) was rejected — less idiomatic and clutters the repo root.

#### Shape

```yaml
# .github/metadata.yaml — read by swift-institute/.github reusable workflow
description: "Swift implementation of RFC 5234: Augmented BNF for Syntax Specifications."
topics:
  - standards          # bare layer tag (Apple redundancy-omission)
  - rfc
  - rfc-5234
  - parsing
  - abnf
homepage: "https://swiftpackageindex.com/swift-ietf/swift-rfc-5234/documentation"
# License is enforced by LICENSE.md presence (not by this file).
# Repo settings default per [GH-REPO-050..053].
settings:
  hasIssuesEnabled: true
  hasDiscussionsEnabled: false
  hasWikiEnabled: false
  defaultBranch: main
```

Required keys: `description`, `topics`. Optional: `homepage`, `settings`. When optional keys are absent, the workflow applies the rule defaults from the `/github-repository` skill.

#### Idempotency contract

The workflow's sync logic is idempotent — running it twice in succession against the same repo produces zero edits on the second run. The convergence criterion is exactly: "next run produces zero edits." This makes the workflow safe to run repeatedly (cron, on-push, manually) without compounding state.

#### Absence semantics

A repo without `.github/metadata.yaml` is logged `skipped: no metadata for {org}/{repo}` and is not touched. **Opt-in by file presence is the rollout's safety property** — a repo that has not been authored against the standard cannot be unintentionally edited by the workflow.

#### Why this shape vs alternatives

| Alternative | Why rejected |
|---|---|
| **Central registry in `swift-institute`** | Forces every metadata change through `swift-institute`'s review queue; metadata becomes a cross-repo dependency on routine package edits. Loses the "edit metadata in the same PR as the code change" property. |
| **Derive from `Package.swift` `description:`** at runtime | `Package.swift`'s description has different semantics (SPM `swift package show` consumes it) and length conventions; conflating them risks breaking either. § 7 Q7 reopens this if explicit-YAML proves friction-heavy after two release cohorts. The Phase-1 generator (§ 4.4) uses `Package.swift` as a *seed* for the YAML draft but does not couple them at runtime. |
| **Just edit GitHub directly** | The status quo. Produces the existing mess (§ 1). |
| **Per-org template with package-name → metadata heuristic** | Loses hand-tunability. Most descriptions need hand-authoring (specification titles, composition phrases, novel domains) that no heuristic recovers. |
| **Path at root (`.github-metadata.yaml`)** | Less idiomatic. GitHub's `.github/` directory is the canonical location for repo-config files; the metadata file belongs there. |

The trade-off: **one extra file per repo (~10 lines of YAML) in exchange for git-tracked, PR-reviewed, idempotent, drift-auditable metadata that converges automatically.**

### 4.2 Centralized reusable workflows

Three workflow files in `swift-institute/.github/.github/workflows/`. All logic lives here; no script duplication.

#### (1) `sync-metadata.yml` — the workhorse (`workflow_call`)

Reusable workflow with inputs `org` (optional), `repo` (optional), `dry-run` (bool, default false). Per in-scope repo:

1. `gh api repos/{org}/{repo}/contents/.github/metadata.yaml --jq .content | base64 -d | yq` → read the YAML (one API call, no clone).
2. `gh repo view {org}/{repo} --json description,repositoryTopics,homepageUrl,…` → read GitHub state.
3. Diff each field; for each divergence, emit `gh repo edit {org}/{repo} --description '…' --add-topic foo --remove-topic bar …` (or log if `dry-run=true`).
4. After write, re-read and confirm convergence; log any persistent divergence (rate-limit retry).

When called with no filters, sweeps all 17 orgs' public non-archived repos. When called with a filter, scopes accordingly. Token (§ 4.5) supplied via `secrets:` block of the call.

#### (2) `sync-metadata-nightly.yml` — the autopilot

Cron `0 4 * * *` (daily 04:00 UTC) + `workflow_dispatch`. Calls `sync-metadata.yml` with no filters and `dry-run=false`. When the run applies any edits, opens or updates a tracking issue in `swift-institute/.github` titled "Metadata sync applied {date}" with a body summarizing the per-repo diffs — for *visibility*, not action. The maintainer sees what auto-converged without having to look.

#### (3) `metadata-pr-preview.yml` — the on-PR previewer (`workflow_call`)

Reusable workflow called by per-repo callers (§ 4.3) on PRs that touch `.github/metadata.yaml`. Calls `sync-metadata.yml` with `repo={current}` and `dry-run=true`, captures the proposed `gh repo edit` invocations, posts them as a PR comment. Acts as a preview before merge.

### 4.3 Per-repo caller workflow (thin, generated)

Each repo carries one tiny workflow at `.github/workflows/metadata.yml`:

```yaml
name: metadata
on:
  push:
    branches: [main]
    paths: ['.github/metadata.yaml']
  pull_request:
    paths: ['.github/metadata.yaml']
jobs:
  on-pr:
    if: github.event_name == 'pull_request'
    uses: swift-institute/.github/.github/workflows/metadata-pr-preview.yml@v1
    with:
      repo: ${{ github.repository }}
    secrets: inherit
  on-push:
    if: github.event_name == 'push'
    uses: swift-institute/.github/.github/workflows/sync-metadata.yml@v1
    with:
      repo: ${{ github.repository }}
    secrets: inherit
```

This caller is **generated** by `Scripts/sync-ci-callers.sh` — the same fan-out tooling that produced the existing 296 caller workflows for the centralized CI rollout. Adding the metadata caller to every public repo costs one entry in the caller-template directory and one re-run of `sync-ci-callers.sh`. No per-repo hand-authoring.

The PR trigger fires the preview comment. The push-to-main trigger (when a PR with `.github/metadata.yaml` changes is merged) fires immediate per-repo sync — convergence within seconds of merge, no waiting for the nightly cron.

### 4.4 Phase-1 generator (heuristic-seeded bulk authoring)

`generate-metadata.yml` — reusable workflow that takes inputs `org` and `repo` (or `org` only for batch). For each in-scope repo:

1. Reads `Package.swift` for `description:` (if present).
2. Pattern-matches the repo name for layer (`-primitives` → `primitives`, `-standard` → `standards`, etc.) and spec-id (`swift-rfc-N`, `swift-iso-N`, `swift-ieee-N`, `swift-incits-N-Y`, etc.).
3. Reads the README's first paragraph as a fallback description seed.
4. Emits a draft `.github/metadata.yaml` covering layer / authority / spec-id / homepage / settings; description is either the `Package.swift` value, the README first paragraph, or — for `swift-rfc-N` / `swift-iso-N` patterns — auto-generated as `Swift implementation of {SPEC-ID}: {Title from spec metadata}.` using a small spec-title lookup table (built from the existing IETF / ISO description shapes already present in the inventory).
5. Opens a PR against the target repo with the draft YAML and a one-line description "Generated metadata draft — review and adjust before merging."

For the ~50 specification-shaped public repos (`swift-rfc-N`, `swift-iso-N`, etc.), the generator produces near-correct YAMLs that need only proof-reading. For the ~20 hand-authoring repos (foundations, novel domains), the generator produces a stub that needs more editing. Either way, the maintainer reviews PRs rather than authoring from scratch — input reduced to "review and merge" for ~70 % of the public surface.

The generator is dispatched manually from the `swift-institute/.github` Actions UI when starting a new wave.

### 4.5 Authentication

Workflows authenticate cross-org via a **GitHub App** (proposed name `swift-institute-bot`) installed on each of the 17 orgs with permissions limited to:

- Repository: **Metadata: Read**, **Administration: Read & Write** (minimum needed for `gh repo edit` to set description / topics / homepage / settings).
- Pull requests: **Read & Write** (for the PR preview comment).
- Issues: **Read & Write** (for the nightly tracking issue).

The App's installation token is minted per-workflow-run via `actions/create-github-app-token`. App ID and private key live as `swift-institute/.github` org-level secrets `METADATA_APP_ID` and `METADATA_APP_PRIVATE_KEY`. Per-repo callers receive the installation token via `secrets: inherit`.

**Alternative**: a fine-grained Personal Access Token with the same scopes, stored as org-level secret `METADATA_TOKEN`. Simpler to set up; tied to a single user's identity (less audit-friendly); rotation is manual. Decision deferred per § 7 Q9.

### 4.6 What the maintainer actually does

| Scenario | Maintainer action | What auto-runs |
|---|---|---|
| Edit existing metadata (typo, new topic) | Open PR editing `.github/metadata.yaml`. Review preview comment. Merge. | PR-preview on open; immediate sync on merge; nightly sweep as safety net. |
| Public flip of a previously-private repo | In the flip-prep PR, add `.github/metadata.yaml`. Flip the repo public on GitHub. | Immediate sync triggers when the PR merges (assuming the repo's `metadata.yml` caller exists; for the just-flipped repo, optionally dispatch sync-metadata-nightly once to converge). |
| Bulk authoring (Phase 1) | Dispatch `generate-metadata.yml` workflow with the org name; review the resulting bot-PRs; merge. | Generator opens N PRs; preview comment per PR; immediate sync per merged PR. |
| Drift response | None routine. If a web-UI edit happens, the next nightly sync reverts it; the tracking issue surfaces what changed. To accept the change instead, edit the YAML to match. | Nightly sync; tracking issue. |

The maintainer never runs a script locally. Every operation is either a PR (reviewed + merged) or a one-click manual workflow dispatch in the GitHub UI.

### 4.7 Boundary with adjacent CI

| Concern | Tool | Lives in | Triggered by |
|---|---|---|---|
| Per-repo metadata (description, topics, homepage, settings) | `sync-metadata.yml` reusable + per-repo `metadata.yml` caller | `swift-institute/.github` + each repo's `.github/workflows/` | PR / push / nightly cron / manual dispatch |
| Org-scoped community-health files (FUNDING.yml, CODE_OF_CONDUCT.md, etc.) | `Scripts/sync-community-health.sh` | `swift-institute/Scripts/` (locally-run) | Manual run by maintainer |
| Per-package CI workflow templates (caller generation) | `Scripts/sync-ci-callers.sh` | `swift-institute/Scripts/` (locally-run) | Manual run after caller-template edits |
| Per-package CI execution (build / test / lint) | reusable workflows + per-repo `ci.yml` callers | `swift-institute/.github` + each repo | PR / push |

The metadata caller is added to the existing `sync-ci-callers.sh` template inventory so every public repo receives both `ci.yml` and `metadata.yml` callers from the same fan-out pass. No new locally-run tooling is introduced; `sync-community-health.sh` and `sync-ci-callers.sh` retain their existing roles unchanged.

Phased. Authorization gates at the moments when GitHub state is written; everything else is local-only or auto-driven by the autopilot.

**Authorization**: none required; investigation produces local files only.

**Exit criterion**: principal review and explicit approval to land the skill (status `PROPOSAL` → `ACTIVE` per [SKILL-LIFE-002]) and to proceed with Phase 1.

**Authorization**: workflow-source PR + caller-template PR are reviewed and merged in `swift-institute/.github` and `swift-institute/Scripts/`; caller fan-out modifies every public repo, which counts as bulk-push per [HANDOFF-023] and requires explicit batch authorization.

### Phase 2 — Heuristic-seeded YAML PRs against `swift-ietf` (canary)

Dispatch `generate-metadata.yml --org=swift-ietf` from the `swift-institute/.github` Actions UI. The generator opens ~45 PRs (one per public repo in `swift-ietf`) with heuristic-drafted `.github/metadata.yaml` files. Each PR's `metadata-pr-preview.yml` automatically posts the proposed sync diff as a comment.

**Maintainer action**: review and merge the PRs as a batch, fixing any heuristic mistakes inline. On merge, each repo's `metadata.yml` caller fires `sync-metadata.yml` → GitHub state converges per-repo.

**Authorization**: bulk PR-opening by a bot account counts as authorship-at-scale; one-time authorization for the generator dispatch suffices. Live GitHub-state writes are gated by the maintainer's per-PR merge action — each merge IS its own authorization for the immediate sync that follows.

**Exit criterion**: 45 `swift-ietf` public repos have `.github/metadata.yaml` merged AND GitHub state converged. Nightly sweep shows zero diff for the org.

Repeat Phase 2 for the remaining ~12 orgs in batches (small body-orgs first, then `swift-standards`, `swift-foundations`, `swift-primitives`). The `swift-foundations` coenttb-personal cohort handled per § 7 Q1's resolution (re-style metadata to Institute conventions).

**Authorization**: per-org generator dispatch + per-PR merge.

**Exit criterion**: all 102 public repos have `.github/metadata.yaml` merged AND GitHub state converged. Nightly sweep shows zero diff ecosystem-wide.

### Phase 4 — Outlier remediation in the same wave

The 47 `coenttb.com` homepage values, the one `experiment/typed-algebra-dsl` default-branch outlier, and the wiki-enabled-by-default settings on ~99 repos are converged automatically by Phase 2 / 3 — the YAMLs specify the correct values, and on-merge sync rolls GitHub state forward. No separate phase needed; just verify the nightly tracking issue from the first post-rollout cron lists these as "edits applied" rather than "diff detected."

### Phase 5 — Public-flip integration (steady state)

Each subsequent public flip (ownership / tagged / property beyond the 0.1.0 cohort, plus future packages) just adds `.github/metadata.yaml` as part of the flip-prep PR. On merge, the per-repo caller fires immediate sync. No separate ceremony.

**Authorization**: none beyond the existing public-flip authorization (the YAML edit and the visibility flip are covered by the same approval).

## § 6. Skill-system encoding

The investigation's findings include a fixed taxonomy, mechanical procedures, and durable per-package conventions — all of which need to land in the skill system, not as a one-off Research doc.

### 6a vs 6b decision

**Decision: 6b — create a new `/github-repository` skill.**

**Rationale**:

- **Volume threshold**: a 6a (extend existing skills) fit would distribute rules as: ~5 to `swift-package` (description format conventions), ~3 to `readme` (description ↔ first-paragraph mirroring), ~1 to `documentation` (homepage ↔ DocC URL), ~3 to `swift-institute` (org / repo settings). That's ~12 IDs across 4 skills. Each affected skill is conceptually a code/content convention skill; bolting GitHub-side rules onto them dilutes their domain. The brief's heuristic (< ~10 IDs across existing skills favours 6a) is exceeded.
- **Cohesion**: GitHub repository metadata is a self-contained domain that does not fit cleanly under any single existing skill. `swift-package` is about Swift-side **package and namespace names**, not GitHub-side metadata. `readme` is about **README content**, not GitHub-side metadata. `documentation` is about **DocC / inline `///`**, not GitHub-side metadata. The closest analog is `readme` (the package's identity-statement at the README level) but the README and the GitHub description are separate artifacts with separate constraints.
- **Future surface**: the brief enumerates branch protection rules, security policy linkage, social preview images, dependabot config, archive policy, repo creation conventions, transfer/archive policy. Each of those is in scope for a dedicated GitHub-repository skill but would compound the dilution if scattered across `swift-institute` / `readme`.
- **Tooling parallel**: the centralized reusable workflows in `swift-institute/.github` (per § 4.2) deserve their own skill so the YAML schema, idempotency contract, dispatch ergonomics, and the per-repo caller convention are durable rules rather than implicit conventions. `Scripts/sync-community-health.sh` and `Scripts/sync-ci-callers.sh` cover *adjacent* surfaces (org-scoped community-health files, per-package CI caller fan-out) and have their own embedded conventions; the metadata workflow family is the third sibling and the cleanest fit for a dedicated skill.
- **Skill-family alignment**: the brief's `github-repository` (per-repo) + future `github-org` (org-scoped community health, sync-community-health.sh's domain) is a clean two-skill family. Encoding repo-scoped rules in a dedicated skill leaves room for the org-scoped sibling without retroactive splits.

**Counter-arguments considered**:

- *"The 6a path is lighter touch and faster to land."* True for the description-format rules in isolation (they could attach to `readme`), but the topic taxonomy + tooling + repo-settings + drift detection are not `readme`-shaped concerns. Forcing them into existing skills introduces shape-class collisions per [HANDOFF-027] (the same rule class reappearing across skills with different framings).
- *"A new skill adds a routing target."* True, but the routing target is unambiguous: GitHub repo metadata operations route to `/github-repository`. The cost of one new skill is paid once; the cost of distributing across four skills is paid every time someone tries to find the rule.
- *"What if `github-repository` overlaps with `readme` on description?"* The overlap is real (README first-paragraph one-liner ↔ GitHub description). Resolution: `readme` owns the README's content; `github-repository` owns the GitHub description and cross-references `readme` for the mirror requirement. Cross-skill cross-references are normal per the skill cluster pattern in [SKILL-LIFE-030].

## § 7. Open questions

Principal review on 2026-04-29 resolved questions Q1-Q6 and Q8 in line. Q7 remains open pending observed friction with the explicit-YAML approach.

### Q1. Coenttb-personal cohort — RESOLVED 2026-04-29

**Resolution.** The coenttb-personal cohort is in active relocation into `swift-institute`; they are Institute repos and proactively adapt to Institute conventions. Treat them as production Institute packages under all rules in § 3 (description re-style per shape A/D, layer tag, no-personal-homepage, etc.).

**Affected repos**: `swift-foundations/{swift-html-fontawesome, swift-html-css-pointfree, swift-html-prism, swift-html-chart}` plus the broader cohort referenced in `coenttb-ecosystem-heritage-transfer-plan.md`.

### Q2. Layer tag canonical form — RESOLVED 2026-04-29

**Resolution.** Apple's redundancy-omission convention. Layer tag is the bare form: `primitives` / `standards` / `foundations` / `components` (no `swift-` prefix). The `swift` tag is similarly redundant and is forbidden. § 3.2 has been updated to reflect this.

### Q3. Homepage default for not-yet-on-SPI packages — RESOLVED 2026-04-29

**Resolution.** `https://swift-institute.org` (no trailing slash). The "leave empty" Apple alternative was rejected in favour of immediate brand presence. § 3.3 has been updated to reflect this.

### Q4. `.github` org repo licenses — RESOLVED 2026-04-29

**Resolution.** `.github` org repos do NOT need their own `LICENSE.md`. GitHub does not support org-level inheritance, but the absence is conventionally unremarkable for community-health repos. **Each PACKAGE repo MUST still have its own LICENSE.md** ([GH-REPO-040] is unchanged for production packages). § 3.4 has been updated to reflect this.

### Q5. `swift-standards/swift-standards` repo — RESOLVED 2026-04-29

**Resolution.** Vestigial. Will be deprecated. The repo was the original umbrella that has since been split into per-domain primitives packages, leaving the umbrella shell with a stale description and a coenttb.com homepage. Out of scope for this metadata standard except that the deprecation should ship its `archive` flag and a final pointing-elsewhere description; metadata for it is not authored under Phase 1.

### Q6. Topic registry governance — RESOLVED 2026-04-29 (subordinate judgement, hybrid)

**Resolution.** Hybrid governance:

1. The `/github-repository` skill ships a curated default registry (initial ~40 domain tags in [GH-REPO-021]).
2. PRs adding a `.github/metadata.yaml` MAY use a domain tag not yet in the registry, provided the PR description carries a one-line justification (`new domain tag: {value} — {reason}`).
3. The skill's registry catches up with newly-introduced tags at the next periodic skill review per [SKILL-LIFE-012] (90 days for implementation skills).

Strict-curated creates bureaucratic friction for an inherently-extensible vocabulary; pure free-tagging produces the existing 49 single-use-tag drift visible in § 1.4. Hybrid lets the registry expand organically while keeping it documented.

### Q7. Auto-derivation of metadata from `Package.swift` — RESOLVED 2026-04-29

**Resolution.** Keep explicit `.github/metadata.yaml`. Auto-derive at runtime is rejected; `Package.swift`'s `description:` is SPM-shaped (≤ 50 chars, consumed by `swift package show`) and decoupled from GitHub's discovery-shaped description (60-120 char optimum). Coupling them would harm one or the other on every edit.

**Compensating mechanism**: rather than reduce the per-package input by erasing the YAML, reduce it by tightening the **template** so the YAML is fill-in-the-blank rather than a writing exercise. § 3.1 has been revised with strict per-class templates and a spec-title lookup table (`swift-institute/.github/spec-titles.yaml`); the `generate-metadata.yml` workflow renders ~70 % of the public surface's descriptions automatically from name-pattern + lookup. The maintainer's input on the description field is reduced to one slot value (a tagline or capability phrase) and only for the L1 / L3 cases where the slot can't be mechanically derived.

### Q9. GitHub App vs fine-grained PAT for cross-org auth — RESOLVED 2026-04-29

**Resolution.** GitHub App, **named `swift-institute-bot`**, with one App per *install scope* rather than one App per *concern*:

| Tier | App | Install scope | Permissions | Used for |
|---|---|---|---|---|
| Cross-org governance | **`swift-institute-bot`** (this decision) | All 17 orgs (and any future ones) | Initial: Metadata r; Administration r/w; PRs r/w; Issues r/w. Permissions accrete as new cross-org concerns arrive — only when actually required, with PR-review gating in `swift-institute/.github`. | Metadata sync (now); community-health propagation, security advisory automation, etc. (later) |
| Umbrella-only concerns | future siblings, e.g. `swift-institute-release-bot` | Only `swift-primitives` / `swift-standards` / `swift-foundations` (3 installs each) | Concern-specific minimum | Release-note generation, package-level helpers (when they emerge) |
| Institute-only concerns | future siblings, e.g. `swift-institute-content-bot` | Only `swift-institute` | Concern-specific minimum | Research / Experiments / Skills indexing, Blog automation |

The shift from "per concern" to "per install scope" responds to the install-effort multiplier (17 orgs × N apps grows fast). Concerns that share an install scope share an App; concerns needing different install scopes get different Apps. The cross-org App's permission scope is contained by three disciplines:

1. **Permissions accrete only when a new concern needs them** — visible in PR review against `swift-institute/.github`; no speculative scopes.
2. **Mint-time token narrowing.** `actions/create-github-app-token@v1` accepts `repositories:` and `permissions:` inputs that scope the *minted token* tighter than the App's full scope. Workflows request only what they need; the App's scope is the ceiling, not the operating scope.
3. **1-hour token lifetime.** Cumulative leak exposure stays low even on the broader App.

The PAT alternative was rejected — its 90-day-default token lifetime, mixed-with-user-identity audit trail, and lack of mint-time narrowing made it a worse fit for steady-state operation across 100+ public repos. Setup-cost differential (PAT ~10 min vs App ~30-45 min) is paid once; ongoing cost differential favours App.

**App registration**: lives at `github.com/settings/apps/<owner>/swift-institute-bot`. No dedicated GitHub repo for the App is needed — the App is a credential registration, not a project. Workflow code that uses the App's tokens lives in `swift-institute/.github/.github/workflows/` (the existing centralized-CI repo). If a "home" page for the bot is desired later, it can live as a small directory inside `swift-institute/.github` (e.g., `bots/swift-institute-bot/README.md`).

### Q8. Drift-audit cadence and enforcement — RESOLVED 2026-04-29 (subordinate judgement; revised for autopilot architecture)

**Resolution.** Subsumed into the autopilot architecture in § 4.2. The "audit" is no longer a separate concern — drift is auto-corrected, not just reported:

1. **Push-to-main → immediate sync.** When a PR modifies `.github/metadata.yaml` and merges, the per-repo `metadata.yml` caller fires `sync-metadata.yml` scoped to the repo. Convergence within seconds of merge.
2. **Nightly cron sweep** (04:00 UTC daily, in `swift-institute/.github`). Reconciles the full public surface against each repo's `.github/metadata.yaml`. Drift introduced via web-UI edits or out-of-band `gh repo edit` is reverted to YAML state automatically. When the run applies any edits, opens or updates a tracking issue summarising what changed — for visibility, not action.
3. **PR-time preview.** `metadata-pr-preview.yml` posts the proposed sync diff as a PR comment so the reviewer sees what sync will do before merge.

Daily cron was chosen over weekly because the autopilot writes drift back automatically — daily means a maximum 24-hour drift window. Push-triggered on every merge to every file was rejected as wasted CI on unrelated commits; the YAML-path-filtered trigger is precise.
